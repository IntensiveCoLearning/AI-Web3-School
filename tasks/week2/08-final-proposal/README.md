## 二、方向选择说明

**主方向：Governance / Coordination——DAO 治理参与率优化**

### 为什么不是纯 AI 问题？

AI 可以做用户分群和内容生成，但它需要**可信、不可篡改的行为数据**作为输入。

Web2 数据的问题：Discord 角色是自填的，Twitter 粉丝可以买，问卷填答可以作假。基于这些数据的用户画像是"用户声称自己是谁"，而不是"用户真正做了什么"。

链上数据解决了这个问题：
- **投票历史**：他是否真的参与过治理，不是说说而已
- **持仓记录**：他在这个 DAO 的 skin in the game 有多少
- **贡献记录**：他做过什么，时间戳和内容不可篡改
- **提案参与模式**：他在哪类议题上表达过意见

AI 如果没有链上数据，只能生成泛化内容；有了链上数据，才能真正做到"对这个人说对的话"。

### 为什么不是纯 Web3 问题？

链上数据是原始事实记录，但它不能自动告诉你：
- **这个人为什么没有投票**——是不关心？没看到提案？不理解内容？还是在等更多信息？
- **什么样的表达方式会改变他的参与决策**
- **同一个提案对不同受众需要哪些不同的角度**

即使拿到所有链上数据，人工分析和内容生成的成本仍然极高。Web3 提供的是可信的行为事实，AI 把这些事实转化成可操作的传播策略。

**两者缺一不可：没有 AI，链上数据是沉默的日志；没有 Web3，AI 的分群依据是可伪造的。**

## 三、问题拆解

### 3.1 参与方

| 角色 | 描述 | 在流程中的位置 |
|------|------|---------------|
| 治理协调者 | 负责提案传播和社区动员的 DAO 管理者 | 发起方，审核 AI 输出 |
| 提案发起人 | 撰写治理提案的社区成员或核心团队 | 内容来源 |
| 活跃选民 | 过去 6 个月有规律投票记录的成员 | 目标受众 A |
| 沉默持币者 | 持有治理代币但从未投票的成员 | 目标受众 B（最大增量） |
| 新贡献者 | 最近 3 个月加入并有贡献记录的成员 | 目标受众 C |
| 流失成员 | 曾经活跃但 6 个月以上无参与的成员 | 目标受众 D |

### 3.2 核心流程

```
1. 输入层
   治理提案文本 + DAO Snapshot Space ID
         ↓
2. 数据获取层（AI 自动）
   Snapshot GraphQL API → 拉取过去 6 个月投票记录
   链上数据 → 持仓快照、贡献记录
         ↓
3. 分群层（AI 自动）
   基于投票频率、持仓量、贡献记录、最后活跃时间
   → 自动识别 4 类受众群体
         ↓
4. 策略生成层（AI 自动，人工审核）
   针对每个群体生成：
   - 传播角度（为什么这个提案和你有关）
   - 内容简报（不同复杂度）
   - 建议渠道（Discord 频道、直接通知、论坛回复）
   - 预测参与概率基线
         ↓
5. 人工确认层（必须由协调者审核）
   内容是否准确？是否有敏感信息？是否符合当前 DAO 立场？
         ↓
6. 传播执行层
   协调者选择版本，手动或授权 Agent 发送
         ↓
7. 反馈追踪层（AI 自动）
   追踪各群体实际参与率，与预测基线对比
   → 更新策略记忆，优化下次分群逻辑
```

### 3.3 AI 作用 vs Web3 机制

| 层次 | AI 做什么 | Web3 提供什么 |
|------|-----------|---------------|
| 数据层 | 理解和解析原始数据 | 可验证的投票历史和贡献记录 |
| 分群层 | 聚类算法 + 行为模式识别 | 链上行为事实作为不可伪造的分群依据 |
| 内容生成层 | 自然语言生成，受众适配 | — |
| 记录层 | 生成审计摘要 | 传播记录可上链存证，不可篡改 |

### 3.4 自动化边界

**可以自动执行：**
- 链上数据拉取和处理
- 用户分群计算
- 传播 brief 草稿生成
- 参与率追踪和报告

**必须人工确认：**
- 最终传播内容发布（内容可能含敏感信息或错误立场）
- 分群策略调整（涉及对社区成员的价值判断）
- 长期策略记忆写入（影响未来所有任务的判断基础）

### 3.5 验证方式

| 验证目标 | 验证方法 | 可操作性 |
|----------|----------|----------|
| 分群准确性 | 与协调者对比，判断 AI 分群是否符合他们对社区的认知 | 访谈 2-3 位 DAO 协调者 |
| 传播效果 | A/B 测试：使用 AI brief vs 统一通知的提案参与率对比 | 需要真实 DAO 配合 |
| Demo 可用性 | Snapshot 公开数据 + 示例提案 → 展示分群和 brief 输出 | 一周内可实现 |

**成功指标：**
> 本项目的成功指标是**治理参与率提升**，而非提案通过率。
> 传播策略只能影响"有多少人参与投票"，不能也不应影响"投票结果如何"——后者取决于提案本身的质量，超出本项目的职责范围。

### 3.6 主要风险

| 风险 | 影响 | 缓解策略 |
|------|------|----------|
| 小 DAO 链上数据不足 | 分群无意义，输出质量低 | 设定最小数据门槛（建议 100+ 历史投票者） |
| AI 放大现有偏见 | 活跃成员获得更多关注，沉默持币者被忽视 | 专门为沉默持币者设计激活策略 |
| 传播内容失真 | 错误的 brief 影响社区信任 | 所有对外发布内容必须人工确认 |
| 数据隐私 | 成员不知道自己的链上行为被分析 | 仅使用公开链上数据；在工具中明示数据来源 |
| 治理操纵嫌疑 | 定向传播被认为是"选择性告知" | 保证对所有成员透明；记录传播范围和策略 |

## 四、项目初步 Proposal

### 项目名称
**GovReach：基于链上行为的 DAO 治理传播优化 Agent**

### 目标用户
- 主要：中型 DAO 的治理协调者（成员 100-5000 人，有定期治理提案的项目）
- 次要：Grant / Hackathon 项目的社区管理者

**用户痛点：**
> 一个 DeFi 协议 DAO 发起了一个调整费率的提案。历史上只有 8% 的代币持有者参与投票。协调者知道很多人"不是不关心"，而是"没有收到合适的通知"。他需要知道：谁是最值得触达的人？怎么跟他们说？但他没有工具，只能群发一条统一通知。

### 真实场景

```
协调者打开 GovReach，输入：
1. Snapshot Space ID（如 aave.eth）
2. 当前提案文本（如"调整协议稳定费率从 2% 到 1.5%"）

Agent 输出：
- 受众分群报告：
  "活跃选民 (23%)：最近 3 个月有 4 次以上投票记录，关注过费率相关提案"
  "沉默持币者 (41%)：持有 >100 AAVE，从未投票，最后链上活动是 4 个月前"
  "新贡献者 (12%)：最近 2 个月有 GitHub/Forum 贡献，首次面对正式提案"
  "流失成员 (24%)：曾经活跃但超过 6 个月无参与"

- 针对每个群体的传播 brief 草稿（待人工确认）
- 建议触达渠道和时间窗口

协调者审核后选择发布版本，
3 天后 GovReach 输出参与率追踪报告，与历史基线对比。
```

### 最小功能（Demo 可实现版本）

**输入：**
- Snapshot Space ID
- 提案文本（手动输入或链接）

**处理：**
- 通过 Snapshot GraphQL API 拉取投票历史
- 基于投票频率 + 最后活跃时间做 4 类分群
- 对每类群体生成传播 brief

**输出：**
- 4 份分层传播 brief（Markdown 格式）
- 简单的受众分群报告（各群比例 + 关键特征）

**不包含（MVP 范围外）：**
- 自动发送功能（所有发布由人工操作）
- 复杂的机器学习模型（规则分群足够验证核心假设）
- 多链数据整合

### 验证方式

**阶段 1（一周内，用于 Demo）：**
- 选取 3 个公开 Snapshot DAO（如 Uniswap、Aave、Gitcoin）
- 用历史提案数据验证分群逻辑
- 展示：相同提案 → 4 份不同 brief，内容差异是否符合直觉

**阶段 2（Hackathon 或后续）：**
- 找 1-2 个真实 DAO 协调者测试
- 用一个实际提案运行全流程
- 对比使用前后参与率

### 主要风险

| 风险 | 应对 |
|------|------|
| Snapshot API 数据不足以区分沉默原因 | 接受局限：只做行为分群，不做动机推断 |
| DAO 不愿意提供真实数据测试 | 用公开数据做 demo，降低合作门槛 |
| 传播定向被认为是"操纵治理" | 文档明确：工具目标是提升参与率，不影响投票方向；所有传播内容透明可查 |

### 可能赛道

- Gitcoin 公共物品资助（治理工具方向）
- ETHGlobal 治理 / DAO tooling track
- 课程内部 Hackathon

### Week 3 下一步

1. 搭建 Snapshot GraphQL API 数据拉取脚本
2. 实现 4 类分群逻辑（规则 based，可迭代）
3. 接入 LLM 生成分层 brief
4. 用 Uniswap 或 Aave DAO 的历史数据跑通完整流程
5. 访谈 1-2 位 DAO 协调者验证分群是否符合他们对社区的认知

## 六、主方向深挖包

### 6.1 典型场景

**场景：Uniswap 费率调整提案**

- DAO 规模：~5 万代币持有者，历史投票参与率约 6%
- 提案内容：将 Uniswap v3 某交易对协议费设置从 0 调整为 0.1%
- 传统做法：在 Discord 发一条通知，论坛贴一个帖子，等待 3 天投票期结束
- **使用 GovReach 之后：**
  - 识别出 2,400 名"曾经投票过费率相关提案但最近 3 个月未参与"的成员——重点激活群体
  - 识别出 18,000 名"持有大量 UNI 但从未投票"的大户——需要低门槛的参与引导
  - 为这两类受众分别生成不同深度的 brief（大户版：2 句话 + 一个链接；前活跃用户版：详细影响分析）
  - 追踪 3 天内两类群体的实际参与率变化

### 6.2 流程图

```
输入
├── 提案文本（What: 提案内容）
└── Snapshot Space ID（Where: 哪个 DAO）
         ↓
数据拉取（AI 自动）
├── Snapshot API：投票历史 / 成员列表 / 历史提案
└── 链上数据：持仓快照（可选）
         ↓
分群计算（AI 自动）
├── 群体 A：活跃选民（高频投票 + 近期活跃）
├── 群体 B：沉默持币者（有持仓 + 无投票记录）
├── 群体 C：新贡献者（近期加入 + 有贡献）
└── 群体 D：流失成员（曾经活跃 + 长期沉寂）
         ↓
Brief 生成（AI 自动 → 人工审核）
├── 群体 A Brief：强调执行细节和影响分析
├── 群体 B Brief：低门槛参与引导，一句话说清楚和他的关系
├── 群体 C Brief：背景解释 + 参与价值说明
└── 群体 D Brief：重新激活，为什么这次值得回来
         ↓
🟡 人工确认（必须）
└── 协调者审核内容准确性 + 授权发布
         ↓
传播执行
└── 按群体选择渠道（Discord 私信 / 频道通知 / 论坛回复）
         ↓
效果追踪（AI 自动）
└── 各群体参与率 vs 历史基线
         ↓
策略记忆更新（人工确认后写入）
```

### 6.3 反例

**❌ 错误做法：全自动定向治理通信系统**

设计一个系统：AI 分析成员数据 → 自动决定给谁发什么 → 自动发送 → 自动更新策略 → 无人工审核。

**为什么这是问题：**
1. 定向传播如果不透明，会被指控"选择性告知"，让有利的信息只到达支持者
2. AI 可能基于历史偏见放大某类声音，系统性地低估某些群体的意见
3. 自动发送错误内容没有撤回机制，对 DAO 公信力的损害不可逆
4. 成员无法知道自己为什么收到特定版本的通知——这是隐私问题也是信任问题

**正确做法：** AI 负责分析和起草，人负责审核和授权发布，所有传播策略透明可查。

### 6.4 关键风险

| 风险 | 严重程度 | 触发条件 | 缓解策略 |
|------|----------|----------|----------|
| 数据稀疏（小 DAO） | 🟠 高 | 投票历史 < 50 条 | 设定最低数据门槛；小 DAO 使用降级模式（仅 2 类分群） |
| AI 偏见放大 | 🟠 高 | 训练数据偏向活跃用户 | 明确要求沉默持币者专项策略；人工审核分群逻辑 |
| 传播操纵嫌疑 | 🔴 极高 | 不同成员收到不同信息但无透明度 | 所有传播策略公开存档；说明分群逻辑 |
| 参与率提升归因困难 | 🟡 中 | 多种传播渠道同时存在 | A/B 测试设计；至少记录工具的触达数据 |

### 6.5 最小验证计划

**目标：** 在一周内证明"基于链上数据的分群输出比统一通知更有针对性"

**步骤：**
1. 用 Snapshot GraphQL API 拉取 `uniswap` space 的最近 20 个提案投票数据
2. 基于规则（投票次数 / 最后活跃时间）完成 4 类分群
3. 用任意一个历史提案文本作为输入，生成 4 份 brief
4. 展示给 1-2 位 DAO 参与者，问：这个分群是否符合你对社区的认知？这 4 份 brief 是否比统一通知更合适？

**成功标准：** 测试者认为分群逻辑合理，且每份 brief 确实比统一通知更有针对性。

## 七、方向 Backlog

以下方向暂时不选，原因如下：

### Backlog A：Payment / Commerce（x402 + CAW）

**不选原因：**
- 课程推荐方向，但与当前积累的模块（C/F/G）衔接度低，临时转向会浪费已有工作
- x402 协议目前生态仍早期，真实可用的测试场景有限，demo 容易流于表面
- 支付闭环的核心价值在于"agent 自主消费 API"，但本项目的 agent 主要消费公开免费数据（Snapshot API），支付需求不迫切
- **后续可能接入点：** 若 GovReach 发展为付费服务，x402 可作为 agent 调用的收费机制，届时自然接入

### Backlog B：Identity / Reputation（ERC-8004 + EAS）

**不选原因：**
- 协议层仍在演化，ERC-8004 尚未广泛采用，构建在上面的产品面临较高的协议风险
- 本方向更适合做 research memo 或 protocol 设计，与 Hackathon demo 形态不匹配
- **后续可能接入点：** 贡献者的 EAS 证明可作为 GovReach 分群的额外数据源，是自然的后续扩展

### Backlog C：Dev Tooling

**不选原因：**
- 竞争激烈（Cursor、GitHub Copilot 等已覆盖大量场景），差异化难度高
