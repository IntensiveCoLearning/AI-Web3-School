今天学习了solidity的发送交易：  
无论是在 DApp 前端中点击按钮，还是在智能合约中用编程方式发起交易，在以太坊的底层看来，都是**向以太坊网络发送了一段经过私钥签名的、符合特定格式的十六进制数据**

# 第一步：构造交易对象

把交易相关的信息组装成一个标准的**以太坊交易对象**。

对于一笔“纯转账（触发 receive）”的交易，这个对象在内存中大概长这样：

```js
{
  "nonce": 42,                  // 你的钱包发出的第几笔交易（防重放攻击）
  "gasPrice": "20000000000",    // 你愿意支付的 Gas 单价
  "gasLimit": "21000",          // 纯转账通常是 21000，如果触发合约可能会更高
  "to": "0x...",      // 接收方
  "value": "100000000000000000",// 转账金额（0.1 ETH，单位是 Wei）
  "data": "0x"                  // 【最关键的一点】附加数据为空！
}
```

-   如果你要调用合约里的某个具体函数（比如 `transfer(address,uint256)`），要将函数名和参数进行哈希和编码，填入 `data` 字段，其中前4字节为函数选择器，其余的为参数
    
-   如果是纯转账，没有指定任何函数，要将 `data` 字段被设置为空字节，在十六进制中表示为 `"0x"`。
    

# 第二步：私钥签名

在签名之前，以太坊客户端（如 Ethers.js 或 Go-Ethereum）会将上述字段按照 **RLP (Recursive Length Prefix)** 编码格式进行序列化。序列化后的原始数据会被进行 **Keccak-256** 哈希运算，生成一个唯一的“交易哈希”。这个哈希值就是该交易的身份指纹。

然后使用发送者的**私钥**对交易哈希进行签名。以太坊使用的是 **ECDSA（椭圆曲线数字签名算法）**，具体曲线为 `secp256k1`。签名过程会产出三个值：`v`, `r`, `s`，这三个值会被附加到交易哈希中，组装成最终的签名交易数据包，即 raw transaction

# 第三步：通过 RPC 广播到网络

拿到这串数据后，钱包会通过 HTTP/WebSocket 请求，调用以太坊节点的标准接口：`eth_sendRawTransaction`

```js
// 底层实际上发送了这样一个网络请求给以太坊节点（如 Infura/Alchemy）
POST / HTTP/1.1
Host: mainnet.infura.io
Content-Type: application/json

{
  "jsonrpc": "2.0",
  "method": "eth_sendRawTransaction",
  "params": ["0xf86b2a8504a817c80082520894..."], // 签名后的 raw transaction
  "id": 1
}
```

节点收到后，验证签名无误，就会把它扔进**交易池（Mempool）**，等待矿工/验证者打包

# 第四步：EVM 执行

当这笔交易被验证者打包进区块，以太坊虚拟机（EVM）开始执行这笔交易。

1、首先看的是 `to` 字段，如果 `to` 地址对应的账户状态里存有 `Code Hash`（代码哈希），节点就知道这不只是一个钱包，而是一个**智能合约**。EVM 会根据这个地址，去底层的数据库（State Trie）里**把该合约编译好的字节码加载到执行环境中**。  
2、代码加载完成，EVM 需要知道执行合约里的哪个函数。这个靠的是交易数据 `data` 字段的前 **4 个字节**，即函数选择器。然后执行相应的函数，或者调用 `receive` 和 `fallback` 函数，如果都没有匹配的，则 revert

# 核心理念的对撞：极简主义 vs. 全栈商业化

在这两个协议的碰撞中，最核心的差异在于它们对协议设计的哲学：

-   **x402（由 Coinbase 和 Cloudflare 主导）：走的是“协议最小化”路线。** 它非常像支付领域的 Unix 哲学，只做一件事：把支付直接嵌入到 HTTP 请求中（复活 HTTP 402 状态码）。它没有账户、没有中间商、无需 API Key，主打开放和无许可，非常适合长尾开发者和去中心化网络。
    
-   **MPP（由 Stripe 和 Tempo 联合推出）：走的是“系统最大化”路线。** 它是专门为 AI Agent 时代的高频交易设计的全栈套件。它不仅解决支付问题，还通过“会话（Sessions）”和流式支付解决了高频并发、风控管理以及法币出入金的问题。
    

# 技术架构与执行机制的核心差异

两个协议在处理具体 Agent 请求时的底层逻辑差异

-   **结算频次与 Gas 消耗：**
    
    -   **x402** 是“单次请求单次结算”。Agent 每发起一次需要付费的 API 调用，都需要在链上（如 Base、Solana）进行一次签名和交易。这在低频场景下很完美，但在高频并发（例如 LLM 流式输出）时会导致昂贵的链上开销。
        
    -   **MPP** 引入了**会话制**。Agent 在前端预先授权一个消费额度（存入资金），随后只需在链下发送轻量级的凭证（Voucher）进行流式微支付，最后再一次性批处理上链。这极大降低了高频调用的开销。
        
-   **支付轨道（Rails）：**
    
    -   **x402** 是纯粹的加密原生（Crypto-native），完全依赖稳定币和区块链基础设施。
        
    -   **MPP** 是轨道中立的。它不仅运行在高 TPS 的 Tempo 链上，最大的杀手锏是**接入了 Stripe 的法币网络**，支持信用卡、钱包甚至先买后付（BNPL），实现了 Crypto 和法币的混合支付。
        

# 安全与合规

-   **x402 的“DIY 模式”：** 资金一旦上链支付便不可撤销（没有拒付）。但开发者需要自己搭建运维脚手架，比如：如何处理退款？如何限制失控的 Agent 疯狂消费？如何做速率限制？这些都需要自己写代码解决。
    
-   **MPP 的“企业级托管”：** 由于深度绑定了 Stripe，MPP 自带了企业级的合规堆栈。诸如 Radar 欺诈检测、PCI 合规、税务处理以及基于共享支付代币（SPT）的可编程支出限制，都已经内置在协议层中。
    

# 并非你死我活的零和博弈

**x402 和 MPP 并不是直接的竞争对手，而是占据了 Agent 经济的不同生态位。**

-   **x402** 适合单次的微支付、无供应商锁定的去中心化架构。
    
-   **MPP** 适合高频的 Agent 会话、需要混合法币支付以及看重企业级风控的商业团队。
    

甚至 Stripe 本身也在其后台同时提供了 x402 和 MPP 独立的两条接入路径，允许开发者根据具体的 Agent 流量特征自由选择

今天学习了无私钥钱包的知识点：  
在交易所创建 Web3 钱包有两种选项，一种是传统的助记词钱包，另一种是交易所托管的无私钥钱包，即基于 **MPC（Multi-Party Computation，安全多方计算）** 技术的钱包。它的核心逻辑不是“没有私钥”，而是“私钥从不以完整形式存在于任何地方”。

### 核心技术：MPC 与私钥分片

传统的钱包会生成一个完整的私钥（或助记词），一旦泄露，资产全丢。而 MPC 钱包在创建时，通过 **分布式密钥生成（DKG）** 协议，直接生成三份独立的“私钥分片”（Key Shares）。

在币安的方案中，这三份分片通常分布如下：

1.  **分片 A（设备分片）：** 存储在你的手机设备中（受手机安全隔离区保护）。
    
2.  **分片 B（云端备份）：** 由你加密后备份到个人的云盘（如 iCloud 或 Google Drive），解密密码由你掌握。
    
3.  **分片 C（币安分片）：** 由币安服务器代为保管。
    

### 交易原理：阈值签名（TSS）

当你需要发起转账时，并不需要把这三个分片拼凑成一个完整的私钥，而是使用 **阈值签名方案（TSS）**：

-   **2/3 原则：** 只要集齐其中 **2份** 分片，就能通过密码学数学运算，“拼凑”出一个合法的交易签名。
    
-   **计算过程：** 你的手机（分片 A）和币安服务器（分片 C）在后台进行一次“秘密对话”。双方各自计算出一部分数据并交换，最终合成一个签名。
    
-   **隐私性：** 在整个过程中，币安看不见你的分片，你也看不见币安的分片，**完整的私钥在物理上从未出现过**。
    

### 为什么它比传统钱包“好用”？

| 特性 | 传统助记词钱包 | 币安无私钥 (MPC) 钱包 |
| --- | --- | --- |
| 上手难度 | 必须抄写并妥善保管 12-24 个单词 | 无需抄词，像登录 App 一样简单 |
| 单点故障 | 助记词丢了或被偷，资产立刻清零 | 丢失一个分片（如手机丢了）也能恢复 |
| 找回机制 | 无法找回 | 通过“云端备份+币安分片”即可恢复 |
| 链上表现 | 普通交易 | 普通交易（比多签钱包更省 Gas 费） |

> 如果手机丢了或者换手机，你只需要：登录你的币安账号（获得**分片 C**），从云端下载并用你的恢复密码解开备份（获得**分片 B**），就可以重新生成分片 A，从而找回钱包控制权。

### 关于云端备份

除了手机本地和币安交易所，需要一个第三方提供的备份存储，比如 iCloud 和 Google Drive

> 这是为了实现 **“抗审查”和“去中心化”**： 如果分片 B 也存在币安的服务器上，那币安就同时拥有了分片 B 和分片 C。这意味着币安可以绕过你，直接动用你的钱。

币安对 iPhone 提供了三种云端备份：快捷备份、iCloud 和二维码。

快捷备份是加密后托管在了**币安合作的安全加密服务商**那里，当你换手机时，只需登录你的币安主账号，并通过**生物识别**验证，币安就会协助你调取这个加密分片来恢复钱包。

iCloud 则是让苹果来托管备份，QR Code 是最硬核的，完全离线，但是图片丢失即无法找回

### 如何在 DApp 中使用？

1、币安 App  
在币安钱包的 Web3 模式下，进入“发现”菜单的 DApp 浏览器，这个浏览器会**自动注入币安钱包**  
2、币安钱包插件  
在浏览器下载币安的官方钱包插件，扫码登录即可  
3、扫码  
没有安装钱包插件，也可以在 DApp 的连接界面选择二维码连接，然后打开币安 App 扫码

今天学习了昨天tc老师提到的账户抽象概念：  
  
账户抽象就是**将现有的“私钥控制的账户”转变为“由代码控制的智能合约账户”**。它让区块链钱包用起来就像现在的网银或支付宝一样顺滑。

# EOA 账户的痛点

-   **绝对权力与绝对风险：** 助记词丢了，资产就永远没了；助记词被盗，资产瞬间被空。
    
-   **用户体验极差：** 每次转账、交互都要签名，还要确保账户里有 ETH 作为 **Gas 费**。如果你想买个链上资产，必须先买 ETH 充当手续费，门槛极高。
    

账户抽象的本质，就是把 EOA 账户的控制权“抽象”出去，让用户直接使用的是一个“合约账户”，但操作起来却像普通账户一样方便。

# 账户抽象的功能

通过引入以太坊的 **ERC-4337** 等标准，账户抽象钱包实现了许多颠覆性的功能：

### 1\. 社交恢复

再也不用死记硬背 12 个助记词了。你可以指定信任的亲友、你的其他硬件钱包、甚至是一家可信机构作为“守护者”。当钱包丢失时，只要**超过半数的守护者签名授权**，就能帮你更换新密钥，找回钱包。

### 2\. Gas 费代付与代币支付

-   **免 Gas 费：** 项目方可以为了吸引用户，替用户支付链上交互的 Gas 费。
    
-   **任意代币支付 Gas 费：** 如果你的钱包里只有 USDT，在过去你无法转账（因为没有 ETH 做手续费）。但在 AA 钱包中，系统可以自动扣除等值的 USDT 作为 Gas 费。
    

### 3\. 交易批处理

在传统的 Web3 项目中，你每个链上操作都要弹出钱包签名一次，极其痛苦。 AA 钱包支持**批量处理**：你可以一次性签名授权“购买、授权、质押”三个步骤；或者开启“会话密钥”，在未来的 2 小时内，该网站内的常规操作自动签名，无需反复弹窗。

### 4\. 自动支付与限额控制

你可以像设置银行卡一样，为钱包设置：

-   每日转账上限（超过额度需要双重验证，防止被盗刷）。
    
-   白名单地址（只能转账给信任的地址）。
    
-   自动续费/定期扣款功能。
    

# 运行机制

ERC-4337 是目前主流的、无需修改以太坊共识层协议的账户抽象标准。它的核心流程如下：

1.  **UserOperation（用户操作）：** 用户发起一个意图（比如“我要转账 100 USDT”），这不再是一个传统的交易，而是一个包含用户需求的结构体。
    
2.  **Bundler（打包器）：** 专门的节点将这些 UserOperation 收集起来，打包成一个真正的以太坊交易。
    
3.  **EntryPoint（入口点合约）：** 这是一个全链唯一的、经过安全审计的中心合约。Bundler 将打包好的交易发给它，由它负责验证每个操作的合法性。
    
4.  **Paymaster（代付方）：** 如果有项目方愿意出 Gas 费，或者用户选择用 USDT 付 Gas 费，就会通过这个角色来处理。
    
5.  **智能合约钱包：** 验证通过后，资金最终从用户的合约钱包中转出。
