## **📝 学习笔记**

### **为什么要学 Context**

LLM 没有自动连接你的数据库、文件系统、API、用户历史和项目文档。它只会基于当前上下文生成输出。如果上下文错了、缺了、过期了，模型再强也会给出不可靠答案。

很多 AI 应用的问题表面看是"模型不够聪明"，实际是 context 设计混乱：

-   把不可信网页当系统指令
    
-   把旧文档当最新规则
    
-   把用户愿望当事实
    
-   把上一次任务的状态泄漏到下一次任务
    

在任何带工具调用的场景里，这都会直接影响安全。模型如果看不到真实工具结果，就只能猜；Agent 如果不知道当前状态、权限和预算，就可能生成越界操作。

### **第一性原理**

> Context 不是简单的长文本拼接，而是一套**信息治理问题**。你要为每类信息标注来源、时效、权限和可信度。否则模型会把"用户说的""网页写的""链上查到的""系统规定的"混在同一层处理。

核心原则：

-   **来源标注**: 每条信息都要标明从哪来的
    
-   **时效管理**: 过期信息要有明确标记
    
-   **权限隔离**: 不同权限级别的信息分开处理
    
-   **可信度分层**: 可信、不可信、待验证信息分层标注
    

### **知识节点**

**1\. Context Window（上下文窗口）**

-   模型一次请求能处理的最大上下文范围
    
-   **窗口越大 ≠ 用得越好**: 长上下文常见问题是"看见了但没抓住重点"
    
-   如果把合约源码、审计报告、治理讨论和交易日志全部塞进去，模型可能被无关内容分散注意力，也可能引用过期段落
    
-   **最佳实践**: context window 应该配合检索、摘要和结构化数据使用。关键状态直接给，长文档按需取，低可信内容隔离标注
    

**2\. Context Engineering（上下文工程）**

设计上下文进入模型的方式，包括：

-   选择数据源
    
-   排序与裁剪
    
-   标注来源
    
-   隔离不可信内容
    
-   决定哪些信息必须每次重新查询
    

一个稳定的工具型 Agent 上下文通常包括：

1.  **系统指令**（角色、约束、安全规则）
    
2.  **当前任务状态**（进行到哪一步，下一步做什么）
    
3.  **可用工具列表**（每个工具的参数、返回值、使用条件）
    
4.  **用户输入**（当前请求，标注来源和可信度）
    
5.  **历史交互摘要**（关键决策，而非完整对话）
    
6.  **实时查询结果**（链上数据、API 返回，带时间戳）
    
7.  **Memory 摘要**（用户偏好、常用配置，但需定期确认）
    

**相关 topic**: Chain-aware Context, RAG

**3\. Memory（记忆）**

-   跨请求保留的信息：用户偏好、历史任务、常用钱包、项目配置、上次分析结果
    
-   **风险**: Memory 可以让 Agent 更顺手，但也容易带来隐性风险
    
    -   系统记住"用户常用 0xabc 地址"，下次就默认它是收款地址
        
    -   记住"用户风险偏好较高"，就放松了高风险交易确认
        
-   **原则**: Memory 不能替代实时授权。用户过去允许某个动作，不代表现在仍然允许
    

**4\. Knowledge Base（知识库）**

-   系统可检索的外部知识库：产品文档、SDK 文档、代码说明、论坛讨论、FAQ、内部 runbook
    
-   **注意**: Knowledge Base 不是让模型"记住"所有内容，而是在需要时检索最相关的部分
    
-   需要配合 RAG 使用，而不是直接塞进 prompt
    

### **在 AI x Web3 中的位置**

Context 是 AI 和 Web3 的**关键桥梁**:

| 场景 | Context 作用 |
| --- | --- |
| 链上数据分析 | 提供实时链状态、合约 ABI、用户持仓 |
| 交易助手 | 当前余额、Gas 价格、滑点设置、权限范围 |
| 智能合约审计 | 源码、依赖库、已知漏洞模式、审计历史 |
| DAO 治理 | 提案内容、投票历史、委托人偏好、规则约束 |

**Chain-aware Context** 是 AI × Web3 的核心概念：Agent 必须知道自己在哪条链上、当前区块高度、交易确认状态、gas 成本等链上上下文。

## **📌 今日核心收获**

Prompt 不只是"怎么问 AI"，而是**任务目标、输入边界、输出格式、失败处理和安全规则**的可执行沟通协议。

* * *

## **🧠 知识节点**

### **1\. Instruction（指令）**

给模型的任务规则，回答五个问题：

-   你是什么角色？
    
-   要完成什么？
    
-   不能做什么？
    
-   遇到不确定信息怎么处理？
    
-   输出应该是什么形态？
    

**关键区分**："解释" vs "执行"

-   研究助手 ✅ 整理资料 / ❌ 假装结论已验证
    
-   代码助手 ✅ 生成 patch / ❌ 默认已通过测试
    
-   交易解释器 ✅ 解释风险 / ❌ 擅自替用户确认
    

**实用四段式写法**：

```
1. 任务目标
2. 可用输入
3. 禁止行为
4. 输出格式和失败格式
```

* * *

### **2\. Few-shot（少样本示例）**

在 prompt 里放少量示例，让模型模仿判断方式和输出格式。

**适用场景**：风格和边界很难一句话说清的任务。  
**示例用法**：解释交易时，不只翻译函数名，还要列出资产变化、风险提示和不确定点 → 放一个好示例 + 一个坏示例。

⚠️ **维护成本**：协议升级、字段变化、业务规则调整后，旧示例可能误导模型。  
💡 **原则**：示例是跟 eval 一起维护的**测试资产**，不是一次写完的素材。

* * *

### **3\. Structured Output（结构化输出）**

让模型按固定结构返回结果（JSON object / 函数参数 / schema 约束字段）。

**为什么重要**：后续系统要处理的是**明确字段**，不是散文。

**Web3 场景示例**：

```
{
  "action": "explain_transaction / prepare_swap / reject",
  "risk_level": "low / medium / high",
  "requires_human_approval": true / false,
  "uncertainties": ["不能验证的事实列表"]
}
```

💡 **核心目的**：让代码能检查、拒绝、记录和回归测试。

* * *

### **4\. Prompt Injection（提示词注入）**

攻击者通过用户输入、文档、邮件或检索内容，让模型忽略原始规则、泄露信息或调用危险工具。

**Agent 场景尤其危险**：模型可能读私有上下文、调用工具、写入系统或触发外部动作。

**应对策略**（不是写"不要被注入"）：

-   把外部内容标记为**不可信数据**
    
-   工具调用前做**参数校验**
    
-   敏感动作强制走 **allowlist + human approval**
    
-   不把**密钥、主权限、不可撤销动作**交给模型
    

* * *

## **🔗 在 AI × Web3 中的位置**

Prompt 处在**用户目标**和**模型行为**之间：

```
用户目标 → Prompt（定义任务和输出格式）
              ↓
         Context（提供可信数据和来源边界）
              ↓
         Model（生成解释或候选动作）
              ↓
         Code（校验 schema 和业务规则）
              ↓
         Guard / Simulation（检查链上影响）
              ↓
         Human Check（确认高风险动作）
```

**Prompt 不应该独自承担安全**。

* * *

## **🛠️ 最小实践（今日作业）**

### **任务：写一个"交易风险摘要"Prompt**

**输入**：

-   交易目标地址
    
-   函数名
    
-   参数
    
-   资产变化
    
-   Simulation 结果
    
-   用户原始意图
    

**要求输出 JSON**：

```
{
  "summary": "string",
  "asset_changes": [],
  "permissions_changed": [],
  "risk_level": "low / medium / high",
  "requires_human_approval": true / false,
  "uncertainties": [],
  "recommended_user_checks": []
}
```

**三组测试用例**：

1.  普通转账
    
2.  无限授权（approve max）
    
3.  目标地址与用户意图不匹配
    

* * *

## **🤔 个人思考**

1.  **Prompt 是接口设计**：以前总觉得 prompt 是"话术"，现在理解它是**软件接口**——定义输入输出、边界条件、错误处理。
    
2.  **安全分层意识**：不能指望 prompt 解决所有安全问题。Web3 里一个错误授权就可能丢资产，所以 **human-in-the-loop** 对高风险动作是必须的。
    
3.  **Few-shot 的维护成本**：以前用 few-shot 时没考虑过版本管理，现在意识到示例和代码一样需要测试和回归。
    
4.  **结构化输出的工程价值**：JSON schema 不只是格式好看，更是让**机器能校验**——这对自动化和安全性都至关重要。

## **📝 学习笔记**

### **为什么要学 LLM**

很多 AI 应用一开始都会说"我们用大模型"。这句话还不够。需要继续追问：

-   模型负责理解用户意图，还是生成内容？
    
-   它只是在回答问题，还是能调用工具？
    
-   它的错误会停留在文本里，还是会进入真实工作流？
    

**LLM 是今天大多数 AI 应用的能力底座**，能：

-   总结文档
    
-   写代码
    
-   生成计划
    
-   抽取结构化信息
    
-   把自然语言转成程序格式
    

**但它本身不是数据库**，不会自动知道外部世界的最新状态，也不能替系统承担事实校验。

### **第一性原理**

> LLM 的核心能力来自**概率建模**：给定前文，预测下一个最可能的 token。

这决定了 LLM 在任何严肃系统里的位置：

-   ✅ 可以帮你理解、归纳、生成和规划
    
-   ❌ 不能单独承担事实来源、权限判断和最终执行
    

**越靠近真实动作，越需要**：

-   外部数据
    
-   确定性规则
    
-   人工或系统校验
    

### **知识节点**

**1\. Token**

Token 是模型处理文本的**基本单位**，不一定等于一个汉字或单词，而是 tokenizer 切分后的片段。

**Token 直接影响三件事**：

1.  **上下文能放多少** - 长文档、代码、JSON 容易塞满
    
2.  **调用成本是多少** - 按 token 计费
    
3.  **模型能不能完整看见关键信息**
    

**💡 实践提示**：

-   代码、JSON、长标识符比普通段落更吃 token
    
-   需要决定哪些信息原样放入，哪些先压缩，哪些交给检索系统按需取回
    

**2\. Embedding**

Embedding 是把文本、代码或其他对象**映射成向量**，用来衡量"语义上是否接近"。

**常用于**：

-   搜索
    
-   聚类
    
-   推荐
    
-   异常检测
    
-   RAG（检索增强生成）
    

⚠️ **限制**：

-   向量相似度只能说明内容接近
    
-   **不能替代来源校验、规则检查和人工判断**
    

**3\. Transformer**

Transformer 是现代 LLM 的**核心架构**，关键能力来自 **attention（注意力机制）**：

-   模型可以在生成时关注输入中的不同位置
    
-   学习词、代码、事实和上下文之间的关系
    

**工程事实**：

-   Transformer 擅长在上下文里找模式
    
-   **不等于拥有稳定数据库**
    
-   可能因为上下文缺失或相似模式误导而给出错误归纳
    

**4\. Hallucination（幻觉）**

Hallucination 指模型生成了**看起来合理、但并不真实或无法验证的内容**。

**常见表现**：

-   编造 API
    
-   错误解释代码
    
-   引用不存在的资料
    
-   把旧版本文档当成当前状态
    

**风险等级**：

-   普通问答：质量问题
    
-   带执行能力的系统：**流程风险**（错误参数、错误权限解释、错误操作建议都可能进入自动化链路）
    

### **在 AI x Web3 中的位置**

LLM 在 AI x Web3 系统里通常承担：

1.  **意图理解** - 把用户自然语言转成结构化目标
    
2.  **内容生成** - 生成解释、总结、代码片段
    
3.  **模式匹配** - 从链上数据、文档、日志中找规律
    

**但永远需要外部校验**：

-   链上状态 ≠ 模型记忆
    
-   合约逻辑 ≠ 模型推理
    
-   资产操作 ≠ 文本生成
    

### **最小实践**

**目标**：用 API 调用一个 LLM，观察 token 消耗、响应时间和输出质量。

**步骤**：

1.  选一个模型 API（OpenAI、Claude、本地模型）
    
2.  准备一段混合中英文的输入文本
    
3.  记录：
    
    -   输入 token 数
        
    -   输出 token 数
        
    -   响应时间
        
    -   输出质量评估
        
4.  尝试让模型解释一段链上交易数据，观察它是否能正确理解

\### 一、 AI时代常见误区——重新审视技术底座与开发价值

1\. **人和AI的协作边界**

\- **合理分工：** 人负责设计方案、评估合理性及验收代码结果；AI负责协助设计、细化方案、按方案执行与自测验证。

\- **核心洞察：** 尽管AI的多Agent编码框架能大幅提升自动化构建效率，但如果开发者自身基础知识与架构能力不足，就无法判断AI端上桌的是“屎山还是珍馐”。

2\. \*\*Web3与Web2的关系

\- **本质联系：** Web3并不是一个完全独立的“新世界”，而是对分布式系统、一致性与安全性提出了更严苛要求的工程演进，其基础架构逻辑依然相通。

\- **系统工程视角：** 大多数Web3项目中Web2能力的占比依然很高（如CEX、Polymarket、DeFi后端等），优秀的Web3开发者首先必须是一个优秀的系统工程师。

3\. **安全的滞后性误区**

\- **核心属性：** Web3系统由于直接与资金交互，安全不是一个可以随时添加的“插件”，而是必须源于设计的核心属性。

\- **纠偏认知：** 必须摒弃“先跑起来再说”（在Web3中设计缺陷意味着资产瞬间归零）和“审计可以补”（大部分审计只是心理安慰剂和合规通行证，安全不可证伪，只有0和1）的盲目乐观。

4\. **核心结论1：Web3工程系统的本质**

\- AI没有降低系统复杂度，它只是加速了代码堆砌。无法解释系统行为，就无法保证资产安全，理解底层逻辑和“安全即理解力”变得前所未有地重要。

\### 二、 Web3常见概念一览——以简单支付系统设计为例

通过对比Web2传统银行支付模型，展示引入Web3稳定币支付时底层架构的变化：

1\. **支付模型演进与区块链本质**

\- **模型替换：** 将Web2中权威的“银行机构”替换为Web3的“区块链网络”（各种L1、L2可类比为不同的银行）。

\- **信任机制转移：** 从信任权威机构转为去中心化信任（数据不可篡改、逻辑即代码公开透明、通过各种共识协议保障）。

\- **区块链本质：** 它不只是一个数据库，更是一个全球共享的、确定的状态机，是利用用户私钥签名进行鉴权、执行智能合约转换状态的计算引擎。

2\. **技术要点与工程实践**

\- **钱包本质与选型：** 钱包是“管理开启资产大门钥匙（私钥）的工具”，扮演身份与资产管理双重角色。开发时需在安全与易用性之间做权衡决策（如多签、硬件、隐私钱包等架构选型）。

\- **钱包安全风险防范：** 需重点防控私钥泄露（资产丢失不可逆）、签名欺骗（需利用EIP712进行签名可视化或禁用eth\_sign）及权限滥用（无限额度授权）。

\- **Web3交易生命周期与关键参数：** 交易历经构造（定义to、Gas、Nonce、Data） $\\rightarrow$ 签名 $\\rightarrow$ 广播（至Mempool）。

\- _Gas：_ 系统的动力源，计算公式为：$\\text{Gas Fee} = \\text{Gas Count} \\times \\text{Gas Price}$。

\- _Nonce：_ 顺序的守护者，用于防止重放攻击和乱序执行，确保状态变更的原子性。

\- _Data：_ 智能合约的指令集，是链上应用交互的灵魂。

\- **交易监听逻辑与挑战：** 实时追踪交易从Pending到Confirmed的状态，捕获智能合约抛出的Event信号以触发后端清分对账。架构上必须具备处理链分叉、区块重组和网络延迟的容错能力。

\- **交易模拟（Simulate）：** 开发中最容易被忽略但至关重要的能力。在提交交易前通过模拟精准预测结果，避免资产损耗和状态不一致，是Debug的前置核心步骤。

\- **Server侧钱包安全防御：** 采用权限拆分（MPC、Safe多签等方案）、“Web2流程 + 风控 + Web3规则”的复合机制，并且坚持只签名可信交易。

3\. **核心结论2：Web3世界对安全的要求极残酷**

\- Web2出错通常是系统问题，数据可回滚、状态可冲正，容错空间大。

\- Web3出错直接导致资产损失，交易不可逆、漏洞即归零，容错成本极高。在Web3世界中，“安全理解力 等于 资产保护力”。

\### 三、 AI时代的变与不变——工具进化下的工程系统本质

1\. **AI改变了什么：效率革命**

\- **Coding更快：** 借由多Agent编码框架大幅减少人在基础代码编写上的精力消耗。

\- **学习更容易：** 跨领域知识获取门槛大幅降低，AI成为全天候技术导师。

\- **基础自动化：** Agent开始接管重复性的系统构建、基础架构部署和初步的代码审计任务。

2\. **AI没改变什么：工程本质的坚守**

\- **系统复杂度不变：** AI无法从根本上消除分布式系统的复杂性，架构设计的严谨性依然是核心。

\- **错误成本不变：** AI生成的错误代码同样会导致严重的资产损失，零容错空间依然没有改变。

\- **安全边界不变：** 最终的审核与责任始终在开发者手中，系统掌控力是开发者的终极防线。

3\. **核心结论**3：人的价值，在于驾驭AI\*\*

\- AI只是一个放大器，它放大的是开发者的架构思维与工程素养，而非取代人类的判断。

\- 当代码编写变得廉价，对系统全貌的洞察力与全局掌控力将变得前所未有的昂贵。未来的核心竞争力在于成为系统底层逻辑的深度掌控者。

\### 四、 架构师能力模型与总结

1\. Web3架构师三位一体能力进化模型：

\- **架构能力 (Architecture)：** 设计高可用、高安全分布式系统，在去中心化与性能之间寻找最优解。

\- Debug能力\*\* (Analysis)\*\*：能够深入底层协议，精准解释每一个逻辑和失败背后深层的技术原因。

\- **基础能力 (Fundamentals)：** 对底层共识机制、密码学原理与链上状态模型拥有深刻且直觉化的理解。

2\. **总结：**

\- **不被AI焦虑驱使：** 它是提高效率的工具，无法替代人对复杂系统的深度洞察与关键决策。

\- **基础能力是唯一护城河：** 深刻理解协议、共识与安全边界在AI时代变得更加关键。

\- **Web3是高标准的系统工程：** 它不仅是编写智能合约，更是对一致性、安全性与分布式协作的极致追求。驾驭技术变革，最终要回归工程本质。
