发现bot与终端的会话上下文互相不通，叫它科普了下怎么个事，正好记笔记里：

你的理解很对！Hermes 的所有数据都存在本地 ~/.hermes/ 目录下面。但 Hermes 确实有机制可以让不同会话互通信息：

\`\`\`

上下文互通机制

1\. Memory（内存）——最重要的互通机制 ✨

Memory 是跨会话、跨平台共享的。我可以在终端里记住你的偏好，你在 Telegram 上问同样的问题时我也知道。

\`\`\`

\`\`\`

| 机制 | 终端 ↔ Telegram 互通？ | 说明 |

|----------------|------------------------|------------------------------------------------|

| Memory | ✅ 共享 | 保存的用户信息、偏好、环境事实，两边都能读到 |

| Skills | ✅ 共享 | ~/.hermes/skills/ 下的技能文件两边都能加载使用 |

| Session Search | ✅ 可搜索 | 终端可以搜到 Telegram 聊过的历史，反之亦然 |

| 当前聊天上下文 | ❌ 独立 | 每个会话有自己独立的聊天记录，互不影响 |

具体来说

Memory — 你让我记住的事实，所有会话都能访问

\- 终端里 /memory save "我的项目在 ~/work/myapp"

\- Telegram 上问「我的项目在哪」→ 也能答出来

Skills — 技能文件存在本地，谁加载谁用

\- 终端里创建了一个 skill（比如代码审查流程）

\- 在 Telegram 上发 /skill github-code-review 也能用

Session Search — 数据都在同一个 SQLite 数据库里

\- 我在终端里可以搜索你在 Telegram 上问过什么

\- 反过来也一样——数据是相通的

但当前对话上下文不通，这是有意设计的——就像你在微信上和同事聊项目，在办公室当面又聊了一遍，两边的聊天记录不会自动合并。如果你想把重要信息从终端带到 Telegram 上，我存到 Memory 里就好啦～ (๑•̀ㅂ•́)و✧

\`\`\`

然后顺便问了下memory，果然就俩md：

存储位置： ~/.hermes/memories/ 目录下的两个纯文本文件：

\- [USER.md](http://USER.md) → 关于你的信息

\- [MEMORY.md](http://MEMORY.md)→ 我的笔记

\`\`\`

机制： 没错，就是你想的那种方式！每次新会话启动时，这两个文件的内容会被注入到系统提示词（system prompt）中，相当于每次开始聊天前我都先"复习"一遍。中间写入的内容要下次会话才生效。

所以你在终端里存了一条 memory，下次在 Telegram 上启动新会话时，我读到同样的 [USER.md](http://USER.md)，就知道你是谁了～

会话启动

↓

读取 ~/.hermes/memories/[USER.md](http://USER.md) + [MEMORY.md](http://MEMORY.md)

↓

注入到 system prompt → prefix caching 生效

↓

开始聊天下回

存储格式就是 § 分隔条目的纯文本，不是 markdown 文档。轻量、跨会话共享、跨平台互通。

\`\`\`

关于我配置的网页爬取工具：

**Firecrawl**：这是目前整个 AI Agent 开发者圈子里最火的开源神器。它的核心本事是：不管你丢给它多么庞大、混乱的网页，它都能瞬间把这个网页**剥离掉所有垃圾广告和花哨的样式，变成干净、规范的 Markdown 文本**喂给大模型。

`DuckDuckGo (ddgs)`：

-   **优势**：**完全免费、不需要任何 API Key**！它直接利用一个 Python 库去后台模拟 DuckDuckGo 的搜索行为。
    
-   **劣势**：网络直连可能会受到环境影响。另外它偏向于“只搜索（Search only）”，它能帮你搜到一堆链接，但如果想点进某个链接去把里面的长文章完整读下来，它的提取能力不如 Firecrawl。

继续昨天的设置：

**Tool Progress Display：**

这个选项决定了你在终端里“监视” Agent 工作时的\*\*可视化颗粒度\*\*。简单来说，就是你想看它思考和行动过程中的多少细节。

一般选all：Agent 每调用一个工具，终端就会打印一行简短的提示

verbose：所有信息全都返回，没什么可读性

new和off：没什么返回，new是调用新工具才返回一行，off是直接黑盒化，都也不是很合理

**Context Compression（上下文压缩）** 就是系统内置的“垃圾回收与记忆摘要”机制。设置太低压缩比较狠，太高的话比较烧token。

    给了一堆选项，但是明显先放本地就够了，反正已经有一个wsl下的ubuntu做隔离。
    
3.  **TTS** 是 **Text-To-Speech（文字转语音）。**这样Agent就可以听咒语了。所谓雷公电母听我号令，张角也不过是掌握了一种调用雷电的方式。
    
4.  Hermes 允许你在同一个 AI 服务商下，**同时塞进去好几个不同的 API Key**。**为什么需要这样？** 因为在跑复杂的 AI Agent 时，Agent 频繁调用工具、思考，极其容易触发高频调用的**频率限制（Rate Limit）**，或者一不小心把某个 Key 的**余额/额度用光了（Exhausted）**。对于做大流量 B2B Token 分发或高频高并发开发的场景来说，这个机制就是他们的救命稻草，能保证 Agent 的业务线永远不掉线。
    
    这个估计后面需要迭代配一下。也有点像那个CC-Switch工具。
    
5.  最大迭代轮数。在 AI Agent 的底层架构中，所谓的“迭代”（Iteration），本质上就是 Agent 跑完一次完整的 **“思考（Thought） -> 行动（Action/Tool Call） -> 观察（Observation）”** 的闭环反馈循环（即业界常说的 ReAct 模式）。 为什么迭代次数是任务完成度的核心？
    
    如果你把最大迭代数设置得太低（比如 10 次），Agent 在第 10 次刚刚把相关的代码文件全部翻找出来、准备开始写修复方案时，系统就会强制掐断它：“你的步数用完了，必须马上输出最终答案。” 这时，你就会得到一个半成品或者胡说八道的回答。**迭代上限，决定了 Agent 解决未知问题时的“思考链长度”和“试错空间”。**
    
    设定过高有什么代价？（Token 与上下文坍塌）
    

-   **烧钱（Token 消耗呈指数级）**：LLM 是没有记忆的。每一次新的迭代，系统都要把**之前所有迭代的思考过程、工具调用记录、长篇的报错信息**全部打包，重新发给大模型。如果你迭代了 100 次，第 101 次请求的上下文可能会高达十几万 Token。
    
-   **注意力涣散（迷失方向）**：上下文太长之后，模型往往会“忘记”你最开始交给它的根本任务是什么，开始在细枝末节的报错里钻牛角尖，陷入死循环。
    

然后根据这个迭代轮次，我去跟AI提了点问题，因为我知道这是一个比较核心的点，能多理解一点迭代，就能多把握一点Agent使用和Agent Coding.提的问题：

但是我的全息盲视又发作了，我们进一步探讨一下：

1.  主流的coding Agent（Codex Claude Code）一般最大迭代轮数在多少，可以自己设置吗？
    
2.  既然迭代轮数消耗token是指数级增长，那么有什么方法，可以令迭代轮数较多的任务拆解为一系列迭代轮数中等的任务呢？这是不是有点类似于Harness Engineering?
    
3.  注意力涣散问题我也见过，有没有一些显著的人类判断力节点？尤其是在工程类任务中，请给我举例。
