今天继续思考AI agent X web3 链上 AI 智能体这一方向  
昨天梳理到，这一方向的核心叙事是赋予 AI 独立控制资产和跨协议执行任务的能力，使其从单纯的“信息处理工具”进化为在数字世界中能“自主交易与协作的经济实体”。  
AI的能力在于 Task-aware 拆解和动态推理，web3的意义在于无需许可的价值传输网关 二者缺一不可，这是方向存在的根基  
今天继续进行深入思考，我试图结合我个人技术背景，思考在现有的模型能力下，实现主要卡在哪  
  
我认为实现难度卡在成本和不确定性两个方面。  
  
一方面，AI agent本身需要消耗token作为燃料，这是一块成本，带AI能力的agent必须要能产生大于token费用的价值，这个AI agent才有持续存在的意义。

另外一方面，AI生成的内容天生充满不确定性，AI给出的决策可能会失真，可能被轻易攻击，如何保证AI在执行危险决策前拿到足够的许可，又让一般决策能自动执行实现便利，这是一块值得思考的trade-off

省流就是：**经济模型（ROI）和安全边界（Security Trade-off）**。  
  
ROI视角：  
如果一个自动化套利 Agent 消耗了 5 刀的算力和 Gas，却只赚回 2 刀，这个系统就是伪需求。因此，目前的破局思路是**高价值密度的场景**。Agent 不能去干“发推特”、“点赞”这种低 ROI 的事情，而必须去执行 DeFi 收益聚合、跨链资金调度、或者高价值数据的筛选与交易。 在这里，“谁承担失败成本”是致命问题。如果 Agent 搞砸了，亏的是用户的钱。我了解到 目前的范式向“Agent 开发者质押保证金”演进——如果 AI 犯错导致损失，从开发者的质押款里扣。这倒逼开发者必须极其严谨地控制 Agent 的行为边界。|  
  
**安全边界：**  
如果 Agent 在执行前置任务时对因果逻辑判断错误，在链上这种不可逆、带有金融属性的环境中就会酿成灾难。  
  
为了解决方便和危险的 Trade-off，我了解到行业内正在探索一种**迭代式生成上下文**的验证机制。 对于低危操作（如查询链上数据、生成投资研报、监控钱包异动），Agent 拥有 100% 的自动化权限； 对于高危操作（如调动大额资金、授权智能合约），Agent 不能直接执行，而是被要求先输出其**任务拆解（Task-aware）的过程和推演路径**。它必须先在沙盒中跑通逻辑，把执行的上下文“迭代”出来。此时，人类（或者另一个专职做风控的小模型）只需要验证这个**因果逻辑链条**是否合理，点击“Approve（授权）”，剩下的繁琐执行再交回给 Agent。这极大地降低了人工协调成本，同时守住了安全底线。

# 2026-05-27

今天认真思考了AI&web3领域下的链上AI agent 方向 ，在ai帮助下整理了一下我学习到的知识  
1\. 问题定位

**Q: 这个方向的叙事中，试图解决的核心问题是什么，能用一句话来说明吗？**

**回答：** 赋予 AI 独立控制资产和跨协议执行任务的能力，使其从单纯的“信息处理工具”进化为在数字世界中能“自主交易与协作的经济实体”。

\### 2. 核心能力解构 (AI vs Web3)

**Q: 这个问题如果没有 AI，是否仍然成立？AI 到底承担什么能力？**

如果没有 AI，链上的自动化早就存在了（也就是智能合约），但它仅仅是死板的“If-Then”逻辑。开发者必须把所有边缘情况都写死在代码里。

\* **AI 承担的能力：** 是面对模糊指令时的\*\*意图理解\*\*、复杂目标的拆解（引入强大的 Task-aware 机制），以及在动态、不确定的链上环境中（比如剧烈波动的流动性池）进行实时决策和\*\*因果推理\*\*。AI 让链上交互从“机器必须读懂代码”变成了“机器能读懂自然语言和复杂策略”。

**Q: 这个问题如果没有 Web3，是否仍然成立？Web3 到底提供什么机制？**

如果没有 Web3，AI 也可以在传统互联网里做 Agent，但它会遇到致命的金融摩擦。你要让 ChatGPT 去帮你买一杯咖啡或投资美股，它很难自己开立银行账户、通过 KYC，也很难使用 Stripe 等传统 API（容易被风控和封禁）。

\* **Web3 提供的机制：** 是\*\*无需许可的价值传输网关\*\*。给 AI 一个加密钱包（私钥），它就瞬间拥有了全球化的支付能力、不可篡改的数字身份（DID），以及一个透明、确定性的执行环境。Web3 解决了 AI 智能体的“财权”和“行动权”问题。

通过这两个基础问题，我们框定了 On-Chain AI Agent 的能力边界：  
\*\*AI 负责处理复杂度和动态决策，Web3 负责提供结算和权限基建。\*\*

1.先准备好测试网的RPC，钱包和测试代币，这里记得私钥千万不能写在会上传的地方，我使用的方式是.env

3.构建交易信息：

tx = {

"from": my\_address,

"to": [w3.to](http://w3.to)\_checksum\_address(TO\_ADDRESS),

"value": [w3.to](http://w3.to)\_wei(0.1, "ether"),

"nonce": w3.eth.get\_transaction\_count(my\_address),

"gas": 21000,

"maxFeePerGas": w3.eth.gas\_price \* 2,

"maxPriorityFeePerGas": [w3.to](http://w3.to)\_wei(1, "gwei"),

"chainId": 11155111,

}

4.执行签名：signed\_tx = w3.eth.account.sign\_transaction(tx, PRIVATE\_KEY)

5.发送交易到节点 tx\_hash = w3.eth.send\_raw\_transaction(signed\_tx.raw\_transaction)

6.使用刚刚发送的交易hash去等待确认 receipt = w3.eth.wait\_for\_transaction\_receipt(tx\_hash, _timeout_\=120)

返回的receipt里面有完整的交易结果'status'为1则代表交易已经完成。

今天总结下本周的学习笔记。之前总结的是理论层面的，今天总结下代码层面的  

\## 这周做了什么

用 Python + [web3.py](http://web3.py)，从零开始跑通了链上数据读取的基本流程。

| 天数 | 内容 | 核心收获 |

|------|------|----------|

| Day1 | 连接以太坊节点，获取最新区块号 | [web3.py](http://web3.py) 的基本用法：HTTPProvider 连接 RPC、is\_connected 检查、eth.block\_number 读链上状态 |

| Day2 | 生成以太坊账户 | eth\_account 库生成地址和私钥，理解地址是公开的、私钥绝不能泄露 |

| Day3 | 查询 ETH 余额 | get\_balance 返回 Wei（最小单位），需要 from\_wei 转换；用 Decimal 避免浮点精度问题 |

| Day4 | 查询 USDT 余额（ERC-20 代币） | 代币不是原生资产，必须通过合约交互；需要合约地址 + ABI 才能调用 balanceOf；decimals 决定精度 |

| Day6 | 监控 Binance 热钱包 USDT 变动 | 轮询模式：循环 + sleep；对比前后余额判断变化；异常处理保证程序不崩 |

\## 几个关键概念

\- **RPC 节点**：你和链之间的桥梁，所有读写操作都通过它。用的是 ZAN 的公共节点。

\- **Wei 和 Ether**：1 ETH = 10^18 Wei。链上存的都是整数（Wei），展示给人看要除以精度。

\- **ERC-20 合约交互**：原生 ETH 用 get\_balance 直接查，代币必须走合约的 balanceOf 函数。需要提供 ABI 告诉 [web3.py](http://web3.py) 函数签名长什么样。

\- **ABI**：合约的接口描述，告诉代码"这个合约有哪些函数、参数是什么类型、返回什么"。

\## 从 Day1 到 Day7 的进步路线

\`\`\`

连接节点 → 生成钱包 → 查 ETH 余额 → 查代币余额（合约交互） → 持续监控（工程化）

\`\`\`

每一步都在前一步基础上加一层：Day1 只是连上了，Day3 开始读数据，Day4 引入了合约概念，Day6 把单次查询变成了持续运行的工具。

# 2026-05-20
<!-- DAILY_CHECKIN_2026-05-20_START -->





ERC-20 是以太坊区块链上最基础、也是应用最广泛的同质化代币标准 。简单来说，它是一套通用的“技术说明书”，规定了在以太坊上发行和运作的代币必须遵守哪些规则。

这里有个要注意的地方，ERC-20规定的是同质化代币也就是说不同的代币个体都是一致的

规定的信息主要包括：

基础信息查询：

 totalSupply ：查询代币的总供应量。

 balanceOf ：查询某个账户里有多少代币。

 name  /  symbol ：获取代币的名称（如 Ethereum）和符号（如 ETH）。

转账与交易：

 transfer ：把代币从自己的账户转给别人。

 approve  /  transferFrom ：允许第三方（比如去中心化交易所的智能合约）在授权额度内，代表你转移代币。

事件记录：

 Transfer  和  Approval  事件：每当发生转账或授权时，会在区块链上留下记录，方便浏览器和钱包追踪 。

erc-20的核心作用就是给出了同质化代币的一个广泛的标准接口，在这一规范下，各个币种的交易能以更低成本实现，同时它更是以太坊乃至整个 DeFi（去中心化金融）世界的通用语言和基础设施

# 2026-05-19

POW共识机制进阶：

1.关于主链的进一步认识：主链不是存储在某个固定位置的东西，主链是直接存在所有人本地的，本地用户要判断两个链哪一个是主链很简单，只要先执行验证，执行验证之后取最长的那一条视为主链，短的那一条直接扔掉。

2.由于主链不基于任何的权威or官方，只基于可验证长度，所以如果两个用户之间发现他们目前参考的主链不一致的时候，只需要直接进行对比就可以了，中间没有信任成本，只有规则。

3.更长的可验证链是主链。这一特性决定：新上链的区块理论上来说是比较有可能回滚的，事实上，在真实情况下，回滚的情况并不罕见，大家都遵循更长的是主链这一个原则，偶尔出现换链就有可能出现，刚打包上的一块，但是新发现了别人的广播，别人比我们多一块，于是当前本地的主链会直接切换，刚刚我们打包上的一块会被直接放弃，交易记录重新回到内存池
