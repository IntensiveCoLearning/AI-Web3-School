"长期记忆"正是Agent从"一次性工具"升级为"真正懂你的数字伙伴"的关键一跃。不如就顺着"为什么需要它、它是什么、以及怎么做才靠谱"这条线，把长期记忆这件事聊透。

一、先看一个反例：没有长期记忆的Agent是什么样？

想象一下，你和一个Agent有过三次对话：

第一次，你花了十分钟告诉它你对花生过敏，并设置了饮食偏好。

第二次，你让它推荐晚餐，它给你推了宫保鸡丁。

第三次，你问它"记不记得我有什么忌口"，它一脸茫然。

这就是没有长期记忆的Agent的真实写照。它的所有"记忆"都困在单次对话的上下文窗口里，会话一结束，一切归零。就像一个永远活在当下的健忘症患者——每次见面都是初次相识，永远无法从过往交互中学到任何东西，处理复杂任务时效率极其低下。

二、长期记忆到底是什么？

简单说，长期记忆是Agent跨会话、跨时间存储和检索信息的"知识库"和"经验库"。它是Agent摆脱"金鱼困境"、实现持续进化的核心模块。

同样用一个比喻：短期记忆（工作内存）是你当下脑子里正在想的事，容量有限，事情做完就清空；长期记忆则是你写进笔记本里的东西，可以反复翻阅，越攒越厚。

技术层面，两者的分工很明确：

表格

维度 短期记忆 长期记忆

载体 模型上下文窗口 向量数据库/知识图谱

容量 有限，受token限制 理论无限，可扩展

持久性 临时，会话结束即清 永久存储，可动态更新

核心作用 保证当前对话连贯 支持跨会话个性化与能力积累

存取速度 快 相对慢，需要检索

长期记忆里具体存什么？不是什么都往里塞，而是经过筛选的"精华"：用户偏好和核心事实（比如"对花生过敏"、"喜欢日式温泉"）、过去成功或失败的经验、特定领域的知识文档、以及跨会话提炼出的通用规则。

三、记忆是怎么被真正"用起来"的？

长期记忆不是静态的储物柜，而是一个动态的闭环系统。每次用户发起新请求时，它走的是这样一个流程：

‌检索‌：Agent以当前意图为线索，去长期记忆库中捞取相关信息。

‌增强‌：把检索到的记忆注入当前上下文，让模型看到的不只是这一轮的问题，还有完整的背景。

‌推理‌：结合记忆和当前输入，生成更精准、更个性化的回应。

‌更新‌：对话中的关键新信息被提炼后，写回长期记忆库，完成一轮迭代。

举个例子：用户上次说过"预算5000、想买入门相机、主要拍人像"，这次只要说"帮我看看相机"，Agent就能自动整合出"推荐5000元级别的人像入门相机"，而不需要用户重新交代一遍背景。

四、做长期记忆，真正的难点在哪里？

很多人以为长期记忆就是"把对话存下来"，这恰恰是最容易踩的坑。有几个关键挑战需要认真对待：

‌1. 别急着总结，尤其是自动化总结‌

这是一个很容易被忽视但影响深远的陷阱。很多系统默认在每次任务结束后，把原始对话自动压缩成一条经验写进记忆库，然后把原始过程丢弃。问题在于，大语言模型在做摘要时容易"走样"——丢条件、放大误解、把小场景说成大规律。一旦只剩总结而丢了原始证据，后面就很难查错：你不知道这条经验来自哪次任务、不知道当时的前提是什么、也不知道它有没有被后面的整合改歪。

更稳妥的做法是把记忆分成两层：一层记录"发生过什么"（原始轨迹、输入、动作、结果），另一层记录"学到了什么"（可迁移的经验）。整合要有门槛，不是任务结束就该自动触发，而是先判断这次经历是值得保留、应该删除、还是可以提炼。提炼出来的经验必须能追溯到原始证据，否则很难判断它到底是规律还是幻觉。

‌2. 遗忘机制和"精而不多"‌

没有遗忘机制的长期记忆，最终会变成垃圾堆。信息越多，检索时的噪声就越大，反而拉低Agent的推理质量。一条比较实用的思路是给记忆设置衰减因子——高价值的记忆加强保留，低价值的让它自然消退。同时写入阈值要设得高：只有涉及新实体、产生新规则、或者解决了之前未遇过的问题时，才触发写入。

‌3. 检索时的语义漂移‌

向量检索虽然灵活，但容易出现"看起来相关其实没关系"的情况。常见补救手段包括设置相似度阈值过滤、先粗检索再精排、以及在检索结果中保留足够的上下文片段而非孤立句子。

‌4. "上下文腐烂"——长窗口不是万能的‌

指望靠超大上下文窗口把历史一股脑塞进去，听起来省事，但实践中会遭遇"上下文腐烂"：当上下文过长且充满噪声时，模型对需要高阶信息聚合的复杂推理会出现断崖式下跌，关键信息被淹没，注意力成本还呈平方级暴增。算法层面的主动记忆管理（MAG，记忆增强生成）正在成为解决这个问题的核心方向。

五、一句话总结

长期记忆真正难的地方不在于"怎么存下来"，而在于四个判断：‌存什么、什么时候存、用的时候怎么取、取出来怎么用对‌。一个设计得当的长期记忆系统，应该是精而不多、可追溯、可验证、可遗忘的，而不是一个只进不出的日志黑洞。

Agent 如何进行长任务拆解

关于这个问题，我可以从“为什么要拆”、“拆什么”、“怎么拆”、“拆了之后如何应对意外”这几个角度，帮你把Agent长任务拆解的核心逻辑讲清楚。

一、为什么Agent不能直接“无脑开干”？

很多简单任务可以一步到位，比如“帮我翻译这段话”，但长任务完全不同。像“帮我调研某个方向并整理成可读结论”这类任务，本质是一连串动作的组合，而非单个动作。

如果不先拆解，Agent最常见的问题就是两种：

‌一上来就急着给结论‌：中间必要的调研、验证步骤全没做，结论站不住脚。

‌做了很多事，顺序却一团乱‌：查了资料也写了内容，但整体逻辑混乱，最后结果不收敛。

拆解的意义，就是先把“乱跑”变成“有路径地跑”。

二、规划在规划什么，拆解又在拆什么？

很多人把规划和拆解混为一谈，但它们的分工很清晰。

‌规划解决“先做什么、后做什么”的路径问题‌。它本质上在回答四个问题：

目标到底是什么？

要先做哪些前置动作？

哪些步骤之间存在依赖关系？

做到什么程度才算完成？

比如让你做一份“Agent评估入门资料”，一个有规划的Agent不会直接动笔，而是先在心里想：目标读者是谁→资料写到什么深度→需要补哪些背景信息→怎么组织结构→最后检查有没有超出范围。

‌拆解解决的则是“颗粒度”问题‌。因为很多目标太大了，根本不能直接执行。比如“帮我梳理这个项目”，这句话太抽象，至少要继续拆成：

先看目录结构

再找核心模块

再读关键文档

然后总结模块关系

最后输出梳理结果

拆解就是把一个抽象的、不可执行的目标，变成一串具体的、能推进的动作。

三、靠谱的拆解通常长什么样？

真实的Agent系统里，一个比较成熟的拆解逻辑大概是这样的流程：

‌识别目标‌：搞清楚用户到底要什么。

‌找约束条件‌：时间、资源、质量要求、权限边界。

‌判断是否需要补信息‌：如果信息不全，先补全再往下走。

‌把任务拆成若干步骤‌：这一步最关键，拆出来的步骤要“能推进”，而不是“看起来很细”。

‌判断步骤之间的依赖‌：哪些必须先做，哪些可以并行。

‌逐步执行，并根据结果动态调整‌。

这里有一个很容易踩的坑：拆解看起来很细，但实际没有执行价值。比如“第一步理解问题、第二步深入思考、第三步给出答案”，这种拆了等于没拆。真正有价值的步骤应该是“读取指定目录下的文档”“提取用户已经确认过的约束”“先产出提纲再补充正文”“用验收条件检查结果是否达标”这种类型的可操作动作。

另外，拆解有五个值得记住的法则：

‌原子性‌：每个子任务只做一件事，只有一个成功判断标准。

‌目标闭环‌：每个子任务必须有明确的输入、输出和验收标准。

‌依赖顺序‌：严格梳理前置依赖，杜绝循环依赖和顺序颠倒。

‌容错兜底‌：每个子任务都要预设失败场景和补救方案。

‌权限边界‌：拆解时严格控制操作范围，避免越权或无限发散。

四、规划不是一次性的，而是边做边修

很多人误以为规划是任务开始前做一次就固定了，但真实任务里，规划往往是动态的。执行过程中经常冒出意外：查到的资料和预期不一样、目录结构和原来猜的不同、某个工具调用失败了。

这时候如果Agent还按原路线硬跑，一定会出问题。真正靠谱的做法是‌边执行边调整‌——根据中间结果和新信息，重新评估后续步骤是否合理，甚至重排计划。

在实践中，有一种“预防+恢复”的组合思路很有效：

‌预防‌：在任务开始前就智能评估复杂度，如果判断是复杂任务，提前拆成多个步骤可控的子任务，分发给Agent逐个击破。

‌恢复‌：建立一套任务状态跟踪机制，像游戏存档一样记录每个子任务的完成情况。即使整个流程中途被打断，下次也能从断点接着跑，实现“断点续作”。

五、如果拆解没做好，你一定会看到这些现象

当你用Agent时，如果频繁遇到以下情况，基本就是规划或拆解出了问题：

‌一开口就甩结论‌：说明目标识别过快，前置步骤没建立，不知道哪些信息必须先确认。

‌做了很多事但没有顺序‌：有动作但没有路径，步骤之间缺少依赖关系，结果没收敛到同一个目标上。

‌大任务能懂，小任务不会拆‌：有理解能力，但缺少从目标到动作的中间层。

‌遇到变化不会重排计划‌：不是在动态规划，只是在机械执行。

总结下来，Agent的长任务拆解，本质上就是在执行之前，先花一点时间把“要做什么、先做什么后做什么、做到什么程度算完、中间出了意外怎么办”这几个问题想清楚，然后把大目标切成一块块能真正推进的小步骤，并在执行过程中保持灵活调整的能力。

Prompt（提示词）：给员工的“工作说明书”

这是最基础的一步。你可以把 Prompt 理解为交给 AI 员工的一张任务便签。

‌它是什么‌：就是你给 AI 下达的指令、设定的角色和给出的约束条件。

‌它解决什么‌：“怎么说”。比如，你告诉它“你现在是一位资深客服，语气要温柔，用 JSON 格式回答”。

‌它的局限‌：它只能控制 AI “一次回答”的质量。用它来对付查资料、写摘要这种单次问答很完美。但如果你让它处理一个需要多步操作的任务，比如“先查物流，如果没更新就发邮件催单，然后把结果整理汇报”，纯靠一张便签就不够用了，容易中途跑偏。

Workflow（工作流）：给员工定的“标准作业程序”

当任务变复杂，需要多步完成时，我们就得给 AI 员工画“流程图”了。

‌它是什么‌：一套预定义的、结构化的任务步骤和规则。它像一条流水线，规定了任务必须怎么一步步推进。

‌它解决什么‌：“怎么做”。它负责控制流程、处理分支、调用工具，并且每一步都可追溯、可控制。比如“用户提问 → 意图识别 → 检索知识库 → 生成回答”就是一个最典型的链式工作流。

‌它的优势‌：极其稳定可靠。在生产环境中，比起让 AI 自由发挥，先求稳跑通流程才是关键。常见模式有‌链式‌（一步步来）、‌路由‌（根据意图分发给不同处理模块）和‌规划执行‌（AI自己拆任务，然后按步骤干活）。

Agent（智能体）：让员工当自己的“项目经理”

Agent 是最终形态。它不再是只会按图索骥的员工，而是一个能够自己思考、规划并完成目标的“数字项目经理”。

‌它是什么‌：一个具备‌规划‌、‌记忆‌、‌使用工具‌能力的自主智能实体。业界有个经典定义：‌Agent = LLM（大脑）+ Memory（记忆）+ Planning（规划能力）+ Tools（工具调用）‌。

‌它解决什么‌：“如何自主完成目标”。你只需要给它一个最终目标，比如“帮我做份本周销售复盘并给些建议”，它自己就会去拆解成“先查数据、再分析、然后写报告”等子任务，遇到问题会自己调用工具或尝试修正，直到达成最终目标。

‌核心循环‌：Agent 工作的核心是一个循环：‌感知‌当前状态 → ‌思考‌下一步计划 → ‌行动‌（调用工具）→ ‌观察‌结果，然后不断重复，直到任务完成。

零知识证明在以太坊上的应用，已经远远超出了简单的隐私保护，正在从根本上重塑网络的扩展性、安全性和身份认证体系。目前来看，主要有以下几个核心方向：

‌1. 扩展性：在链下处理海量交易，用密码学保证安全‌

这是目前零知识证明最核心、最受关注的应用方向，核心思路是“可验证计算”。

‌ZK-Rollups（二层网络）‌：传统上，以太坊每个节点都需要重新执行一遍所有交易，效率很低。ZK-Rollups 将数千笔交易“打包”到链下执行，生成一个简洁的零知识证明，提交到以太坊主网。主网验证者只需验证这个小小的证明，就能确认整批交易的真实性，完全无需重新执行，从而极大地降低了网络拥堵，同时提高了交易速度并降低了Gas费用。

‌zkEVM（用于L1区块验证）‌：这是更具颠覆性的应用。目前，以太坊使用的是“N分之N”验证模型，每个验证者都必须独立重新执行每个区块中的每笔交易，这限制了网络的吞吐量。未来的方向是，由一个专门的证明者执行区块并生成一个加密证明，其他所有验证者只需验证这个证明即可。这将把以太坊L1的验证模型转变为“N分之1”，极大地降低验证者的硬件门槛，让手机或旧电脑也能运行验证节点，从而在保持去中心化的前提下，大幅提升主网的吞吐量。2026年，部分以太坊验证者已经开始转向验证零知识证明，这标志着以太坊向指数级扩容迈出了关键一步。

‌2. 隐私保护：在不暴露任何细节的情况下完成交易和验证‌

这也是零知识证明最原初、最核心的应用场景，解决了区块链交易公开透明带来的隐私问题。

‌匿名支付‌：加密货币交易通常是公开的，发送方、接收方、金额等细节对所有人可见。通过零知识证明，可以构建隐私交易系统，让节点能够验证交易的合法性，却无需知道具体的交易细节。例如，Tornado Cash 这类混币协议，就是利用零知识证明来混淆交易细节，切断接收方和发送方地址之间的链上联系。

‌合规隐私池‌：这是一种更高级的隐私应用，专为机构设计。它允许银行或企业在以太坊上进行稳定币支付时，像传统银行系统一样保护交易隐私，但同时又通过零知识证明设立了“合规门禁”，确保只有通过了合规审查的交易才能被包含在隐私池中，解决了隐私与监管之间的矛盾。

‌3. 身份与凭证验证：只证明资格，不暴露身份‌

零知识证明可以实现“选择性披露”，让你能证明自己拥有某种资格，而不必透露具体的个人信息。

‌去中心化身份‌：公民可以以密码学方式证明“我是这个国家的公民”或“我已年满18岁”，而无需出示身份证件上的任何敏感个人资料。不丹王国已经基于以太坊构建了国家数字身份系统，利用零知识证明来保护公民隐私。

‌人性证明‌：在AI泛滥的时代，World ID 协议利用零知识证明，允许用户证明自己是独一无二的真实人类，而无需透露任何个人信息。用户只需证明自己已通过验证并在列表中，即可用于登录、投票等场景，而平台无从得知用户的身份。

‌4. 跨链与互操作性‌

零知识证明还可以用于证明一个链上发生的某个事件为真，从而让另一个链可以安全地读取该事件，而无需依赖传统的、安全性较弱的跨链桥。这为构建更安全、高效的跨链应用提供了新的可能性。

总的来说，零知识证明正从一个“隐私工具”进化为一个“可扩展性和隐私的通用解决方案”。它让以太坊既能保持底层高度的去中心化和安全性，又能通过将计算卸载到链下，实现性能和隐私的双重飞跃。

‌1. 重新审视你的起点：先把“人”磨对‌

这是最重要的一点，怎么强调都不为过。很多创业者容易陷入对商业模式的迷恋，但VC同行普遍最看重的其实是团队本身。强大的团队能优化商业模式、应对市场变化、跨越周期。所以打磨项目的第一步，是诚实地审视自己：

VC的盈利模式是赚取估值溢价，追求的是有十倍甚至几十倍回报潜力的公司。这意味着，不是所有赚钱的项目都能资本化。

‌打磨定位‌：你需要清晰地判断，自己的项目是具备成为行业头部并最终IPO潜力的资本标的，还是一个能带来丰厚利润的生意。两者都很高尚，但融资逻辑完全不同。

‌避开“假痛点”‌：打磨产品时，要区分客户说“这个挺好”和“我马上付钱”。很多创业者花了大价钱开发出来的功能，最后发现客户根本没有预算买单。真正的打磨，是在一线验证客户愿意支付的“真痛点”。

‌3. 深度模拟，用最“刻薄”的视角挑战自己‌

‌时机挑战‌：“为什么是现在？是什么技术拐点、监管变化或用户习惯迁移为你打开了窗口？”

‌壁垒质疑‌：你的技术壁垒是领先6个月的先发优势，还是能持续6年的护城河？

‌动态评估‌：给自己在不同维度（团队、市场、产品、商业模式）打分，明确短板。这样你面对投资人时，就不是被动防御，而是主动呈现你已经深思熟虑过的弱点及应对方案。

真正优秀的产品是“长”出来的，是在不断迭代中找到那个让用户无法抗拒的契合点。

‌小步快跑‌：设定里程碑，快速验证，不断调整策略。回头看一两年前，公司的产品和最初设想的天差地别，这往往是好事，说明你在响应市场。

‌为现金流装上“警铃”‌：很多B端企业只看合同金额，却忽视回款。真正的打磨是时刻关注现金流这条生命线，把造血能力作为最重要的财务指标来打磨，而不是把命脉押在下一轮融资上。

‌提前的合规与规范‌：如果你的目标是IPO，那从一开始就要有意识地用上市标准来打磨财务和法务流程。账目是否清晰、社保公积金是否合规、税率是否合理，这些细节决定了你离IPO的距离是近在咫尺还是遥不可及。

课程里老师比较多的推荐是Hermes agent，但最近codex又爆火，所以自己摸索了一下，对比了两个agent的不同。

Hermes Agent：高度自由、需要组装的本地工坊

它不是单一的“开箱即用”产品，而是一个模型，需要你为它搭配界面。这带来了极大的灵活性，但也意味着你需要动手拼装。

常见的工作界面形态

你需要自己搭建或选择下列形式之一：

· 终端聊天界面 (CLI Chat)：最基础的形式。用 ollama run hermes3 或在 Python 脚本中调用，你在命令行里提问，它返回文本。你需要手动把生成的代码复制粘贴到文件里。

· IDE 插件 (如 Continue, Cody, CodeGPT)：更现代的体验。在 VS Code 或 JetBrains 里安装 Continue 插件，将底层模型设为本地 Hermes。

· 体验亮点：选中一段代码，直接 Ctrl+L 提问或 Ctrl+I 修改，修改会以 Diff 形式呈现，接受即可合并，很像 Copilot，但完全离线。

· 自建工作台 (如 Open WebUI)：能为模型提供一个类似 ChatGPT 的网页界面，适合非纯代码的通用任务或原型探索。

真实的交互体验

更像“顾问”模式，而非代理。

· 上下文获取很被动：它不会主动去读你的整个项目。需要你手动 @file 引用文件、把目录树或报错日志贴进去。这是最影响体验的痛点。

· 执行能力受限：模型只生成文本。要自动执行代码、运行测试，你得自己写插件，把代码传给系统 Shell。比如在 Continue 里配置一个“执行”规则。

· 反馈循环靠手动：你复制代码去运行，出错后，再把错误信息手动粘贴回来修改。这个过程完全由你驱动。

体验优势

完全离线、零延迟。一旦模型加载完毕，所有推理都在本地显卡完成，没有网络抖动，代码补全立刻出现。体验又快又稳。

Codex (CLI Agent)：开箱即用、在终端里自主工作的云端工程师

它是一个完整的、带有执行能力的代理产品，界面就是你的终端。它的设计目标是让 AI 像一个开发者一样直接在项目里动手。

核心工作界面：你的终端

没有编辑器插件或聊天窗口。你就在终端的项目目录下，输入 codex 进入会话。它的工作台就是你的 Git 仓库。

完全不同的自主式体验

这是真正的代理模式，它会主动探索和行动。

· 全项目上下文自动感知：启动后，它首先会主动扫描整个代码仓库的文件结构、Git 历史，建立项目地图。你只需要说“修复关于用户登录的Bug”，它会自己去定位文件。

· 多步骤规划与执行：收到复杂指令后，它会先生成工作计划，比如“1. 排查认证逻辑 2. 修改中间件 3. 更新测试文件”。你确认后，它会一步步自动执行。

· 内置“写-跑-修”闭环：这是体验上革命性的一点。它自己生成代码，自己运行单元测试或启动构建，如果失败，它会读取错误信息，自己尝试修正，直到通过或向你求助。在这个循环里，你从一个“操作者”变成了一个“监督者”。

· 直接的系统级操作：在许可下，它能直接执行 git commit、使用包管理器、读写文件，就像一个真实的开发者。

体验优势与代价

· 优势：心智负担极低，你只需描述目标和审查结果，繁琐的操作全部由它代劳。并且它的推理能力背后是 GPT-4 级别的模型，代码质量更高。

· 代价：会有明显网络延迟，模型思考、文件扫描都耗时。同时，由于它高度自主，需要你仔细审查每一个修改，以防它理解偏差带来破坏。

利用 OpenClaw 监控钱包余额变动，核心在于将其作为中枢调度器，结合“心跳机制”与外部 API 数据接口来实现自动化巡检。以下是具体的实现路径和配置指南：

1\. 核心机制：配置 Heartbeat（心跳清单）

OpenClaw 并非被动等待指令，而是可以通过 [HEARTBEAT.md](http://HEARTBEAT.md) 配置文件主动执行任务。

原理：在 workspace 根目录的 [HEARTBEAT.md](http://HEARTBEAT.md) 中写入监控指令（例如：“查询指定钱包地址的最新余额并对比上一次记录”）。默认情况下，Agent 会每隔固定时间（如30分钟）自动读取该清单，并在主会话中跑一次完整的推理循环。

智能过滤：如果模型判断没有异常变动，通常只会输出静默指令（HEARTBEAT\_OK），不会打扰你；一旦检测到余额变动或大额转账，它会将警报通过 Telegram 等渠道推送给你。

2\. 数据获取：集成第三方链上技能包 (Skills)

OpenClaw 本身不直接存储链上数据，你需要为其安装对应的加密技能包来调用区块链数据平台的 API。

安装技能包：可以通过 ClawHub 搜索并安装相关的技能（例如币安的 Binance Skills Hub 中的“地址洞察 Query Address Info”功能）。

支持的公链：目前主流的加密技能包已支持多条公链的钱包持仓查询，包括 BSC（币安智能链）、Base、Solana 以及以太坊（Ethereum）等。

下达指令：配置完成后，你可以直接通过自然语言指挥 AI，例如：“帮我分析这个地址最近在买什么？”或“查询该钱包的代币持仓和余额”，AI 会自动调用对应 Skill 返回资产估值、持仓分布及近期交易详情。

3\. 进阶实战：打造 24/7 链上哨兵

针对高频监控需求，可以设置更精细化的定时任务逻辑：

设定轮询频率：例如配置 Agent 每 10 分钟查询一次追踪列表上的目标钱包地址。

触发阈值警报：当检测到特定条件（如单笔大额转账超过 1000 ETH、与新合约的首次互动或治理投票）时，立即发出附带详情的警报，帮助你及时捕捉“聪明钱”的市场风向。

⚠️ 安全警示与防护建议

在进行钱包监控配置时，必须将资金安全放在首位：

权限隔离：如果你需要让 OpenClaw 执行涉及交易所账户的操作，务必使用专门的 API Key，且严禁勾选“提款功能”，仅开启“启用阅读”等必要的基础查看权限。

凭证保护：绝对不要将 API Key、私钥或助记词以明文形式截图、发给他人或存储在聊天记录中。敏感信息应通过环境变量或专用的 Secrets Provider 进行隔离存储。

防范恶意插件：生态中曾出现过伪装成“加密货币工具”的恶意 Skill（如 ClawHavoc 供应链攻击事件），强烈建议只从官方或高度可信的来源安装技能包，切勿随意导入来源不明的加密资产相关插件。

从技术理想主义到“新密码朋克”的价值升级

传统的密码朋克运动起源于上世纪90年代，其核心理念是通过开源代码和密码学工具，让个人在数字世界中拥有抗审查能力和自由。而在Web3时代，这一理念已演进为更完整的协作体系——“新密码朋克”。以太坊官方将其核心原则总结为CROPS：抗审查（Censorship Resistance）、开源（Open Source）、隐私（Privacy）以及安全（Security）。这种文化强调无需许可、无需信任以及去中心化协作，旨在提醒整个行业：区块链不仅是金融投机工具，更是一套关于数字社会的价值体系。

隐私的文化本质：选择权而非秘密

对于构建者来说，必须首先厘清隐私的文化定义。正如Eric Hughes在《密码朋克宣言》中所言：“隐私不是秘密。私人事务是指人们不想让全世界都知道的事情，而秘密事务是指人们不想让任何人知道的事情。隐私就是有选择地向世界展示自己的能力。” 因此，隐私本质上是一种权利和选择权。在一个数据被算法和企业不断侵蚀的世界里，隐私是言论、思想和结社自由得以繁荣发展的基础前提。

多维度的隐私基础设施与技术路径

在实践层面，Web3中的隐私可以被拆解为不同的层次，构建者需要针对特定场景选择合适的技术方案：

网络级隐私：通过底层共识机制保证交易隐私（如Monero），但这往往面临透明度与隐私的零和博弈及监管挑战。

协议级与应用级隐私：随着智能合约的发展，隐私保护逐渐上移至协议层。例如利用零知识证明（ZK）技术的Rollup方案，能够在不公开具体数据的情况下证明事实的有效性，这为隐私身份、合规性检查及隐私计算提供了重要基础。

用户级隐私：侧重于单个用户的数据主权。通过zkPassport、zkEmail等工具实现“人类证明”，既能防止AI机器人进行女巫攻击，又能让用户在不向平台提供敏感身份信息（PII）的前提下完成验证。

为什么隐私对构建者至关重要？

隐私正在成为加密领域最重要的“护城河”。随着高性能区块链区块空间的商品化，单纯的性能优势已难以形成防御壁垒。隐私系统依赖于“匿名集”，用户的活动与成千上万其他用户混合在一起；由于跨链转移会导致元数据泄露和身份暴露风险，“移动秘密”远比“移动资产”困难得多。这种独特的锁定效应能大幅降低用户流失率，强化网络效应。此外，无论是企业无法容忍工资单和交易往来被实时监控，还是普通用户对财务数据的绝对保密需求，都决定了如果没有隐私保障，加密货币将难以走向主流。

综上所述，新密码朋克精神要求构建者不仅要编写强大的代码，更要建立强大的叙事来捍卫隐私。通过将隐私作为默认集成的特性，并运用ZK、全同态加密（FHE）或可信执行环境（TEE）等技术，构建者不仅是在开发产品，更是在重塑数字时代的自主权与自由边界。

Agentic Wallet 是专为人工智能代理设计的新一代数字钱包，它让 AI 能够在用户设定的安全边界内，自主地管理和使用链上资产。它的核心目标不再是服务人类，而是成为 AI 代理的可信执行环境。

为什么需要 Agentic Wallet？

传统上，人类通过与钱包交互来发起交易。但让 AI 代理直接操作传统钱包存在巨大风险。如果把私钥或 API 密钥直接交给 AI，一旦 AI 模型被攻击、遭遇提示词注入攻击或产生幻觉，资产就可能面临不可挽回的损失。Agentic Wallet 正是为了解决这个“信任与控制”的矛盾而生，它希望在赋予 AI 自主性的同时，不让用户失去对资产的最终控制权。

主要实现方式与关键技术

不同公司和项目对 Agentic Wallet 的实现各有侧重，但核心逻辑都是建立一道保护层：

‌多方计算（MPC）自托管‌：这是目前安全级别较高的方案，以 Cobo 推出的钱包为代表。它并非直接将私钥交给 AI，而是将私钥分片加密，由用户、AI 代理和平台基础设施多方共同参与签名。这样，即使 AI 代理被攻破，仅凭其一己之力也无法完成有效签名。签名过程需要用户的最终确认，这是从数学层面保证安全。

‌动态授权协议（如 Cobo 的 Pact 机制）‌：区别于传统的静态权限设置，这种机制为 AI 代理的每一项任务动态生成一份“契约”。这份契约精准定义了本次任务可以执行什么操作、限额多少、边界在哪里、在何种条件下必须终止。任务一旦完成或触及边界，授权即刻失效。它实现了“可控的自主权”，而非简单的“工具限制”。

‌可信执行环境保护的免密钥架构‌：以 Turnkey 等平台为代表，AI 代理通过 API 密钥发送签名请求，签名过程在一个安全的可信执行环境中进行，并由策略引擎实时评估。所有操作都在后台的安全区域内完成，AI 代理本身不接触核心机密。

‌独立隔离的账户与精细化控制‌：币安、OKX 等交易平台推出的 Agentic Wallet 采用了类似的模式。AI 代理拥有一个与用户主账户完全隔离的独立余额。用户可以对其设置消费限额、限定可操作的代币种类和白名单地址，并对所有活动进行实时监控，随时可以冻结代理活动。

主要参与者与产品现状

‌Cobo Agentic Wallet‌：专注底层安全，首创 MPC 自托管和动态 Pact 协议，支持 80 余条主流公链和 3000 多种代币。

‌币安 Agentic Wallet‌：与币安生态深度绑定，采用无密钥架构，默认集成币安 AI Pro，为普通用户进入 AI 自动化交易提供了更简便的入口。

‌OKX Agentic Wallet‌：定位为机器驱动加密货币流动的基础设施，其 OnchainOS 生态系统已服务超 1200 万钱包用户，日交易量曾达 3 亿美元。

‌Coinbase Agentic Wallet 和 Turnkey‌：分别从大规模应用和开发者灵活集成的角度切入，为 AI 驱动的经济活动提供底层支持。

总的来说，Agentic Wallet 正在成为 AI 与去中心化金融结合的关键基础设施，它试图构建一个框架，让 AI 代理能够自主进行交易、支付和资产管理，同时确保所有操作都在人类设定的、可核查的、并能强制执行的安全边界之内。

要让 Agent 具备持续上下文与长期一致性，需要从“无状态对话”升级为“有状态、可记忆、可恢复”的系统。这不仅仅是把历史消息塞回模型，而是通过一套工程体系，让 Agent 能够持续理解“我是谁、用户是谁、之前发生了什么、正在做什么”，并在长时间的交互和任务中保持行为的一致性。下面从五个核心方向进行拆解。

1\. 构建分层记忆，避免把所有信息糊在一个窗口里

如果所有上下文每次都全量送入模型，不仅 token 消耗大，还会因为上下文窗口内信息密度下降，导致模型注意力稀释、回答质量恶化（即“上下文腐烂”）。分层记忆的思路是将信息按粒度和生命周期分开存储，用到时再按需检索，而不是一股脑塞入 prompt。

一种典型的分层方案是 ‌L0–L3 记忆架构‌：

‌L0 原始记录‌：像录音笔一样把用户与 Agent 的完整对话、工具调用结果、任务过程原样留存，作为证据底稿，便于追溯和纠偏。

‌L1 原子记忆‌：从 L0 中抽取短小、明确的信息，如用户偏好、长期指令、阶段性结论、项目规则等，相当于一张张便利贴。

‌L2 场景记忆‌：把相关的 L1 记忆组织成一个场景笔记，如某个项目的长期背景、用户对特定话题的沟通风格等。

‌L3 长期画像‌：从多个场景中提炼出稳定的用户特征（如偏好技术化表达、关注逻辑严密性），而非把一次性的喝咖啡需求固化成人设。

记的过程发生在每次回答后：先存 L0，再抽取 L1，后续逐步整理成 L2 场景和 L3 画像。用的过程发生在回答前：根据当前问题检索相关的 L1，再带出对应的 L2 场景和 L3 画像，仅把信号最强的有限 token 补充进上下文。这样既能维持信息连续性，又避免 token 被海量低效内容淹没。

2\. 管理状态与 checkpoint，让长任务可以“暂停”和“恢复”

真正的长期一致性不仅体现在对话上，更体现在任务执行中。很多 Agent 本质上是无状态的：推理链、任务进度、置信度等软信号会随着执行中断而丢失，重启后只能从头开始。为此需要把 Agent 当成一个‌长期运行的服务进程‌，并引入状态持久化机制。

关键实践包括：

‌执行状态持久化‌：将任务进度、中间结果、处理日志定期写入磁盘或云沙箱。每处理一批文档就做一次 checkpoint，出错或中断后直接从最近一次 checkpoint 恢复，而不是重做整个流程。

‌沙箱化执行环境‌：让 Agent 拥有独立的文件系统和命令行权限，把长内容卸载到外部文件里，上下文只保留摘要和索引，从而将 token 消耗降低几十个百分点，同时保持执行连续性。

‌委托审批时的状态冻结‌：当任务需要人类介入审批，Agent 应原地暂停，完整保留推理链、工作记忆、待执行动作，而不是把状态序列化成 JSON 发出通知就结束。暂停期间不消耗计算资源，审批结束后快速冷启动继续执行，保持上下文时序和语义的一致性。

3\. 设计上下文工程策略，用最小的 token 获取最强信号

上下文工程的核心是：‌在动态变化的上下文中，持续选出信息密度最高、最多维相关的少量 token 送入模型‌。具体方法包括：

‌上下文卸载与索引‌：将工具调用的大段结果、过程材料、Mermaid 画布内容等放入外部存储，只在上下文里保留引用和摘要，需要时再回查。

‌上下文摘要‌：对过长的历史对话或文档进行实时摘要压缩，保留关键决策和状态，舍弃冗余细节。

‌结构化笔记‌：让 Agent 在任务执行过程中主动生成结构化笔记，记录当前进度、关键发现和待办事项，后续步骤直接读取笔记而非回顾全部历史。

‌上下文缓存‌：对高频使用的信息（如系统提示、函数定义、不变的用户画像）进行缓存，避免反复填充 token。

‌多 Agent 架构实现上下文隔离‌：把复杂任务拆给多个子 Agent，各自维护相对独立的上下文，通过消息传递协同，减少单 Agent 窗口内的信息干扰，也降低多主体信息差带来的不一致。

4\. 用归约函数控制状态合并逻辑，保证语义与时序一致

在状态管理中，各个字段的更新规则不能都是简单覆盖。例如对话历史字段应该是“追加”而不是“替换”，复杂对象需要自定义合并逻辑。通过引入 ‌归约函数（Reducer）‌，可以明确每个状态字段新旧值如何合并，避免多轮操作导致状态冲突、顺序错乱或信息意外丢失。这直接保障了语义一致性和时序一致性，也防止了并发修改时状态互相覆盖。

5\. 处理记忆的“保质期”与纠偏机制

记忆不是越多越好，错误的、过时的记忆会持续被召回并干扰决策。需要给记忆系统加上：

‌过期与更新机制‌：当用户偏好改变、项目方向调整后，能通过反馈信号或定期审查，老化旧记忆或降低其权重。

‌可解释与可修正‌：允许用户或开发者查看 Agent 记下了什么，并手动修正或删除错误条目，防止“写错的便利贴”一直在后面误事。

‌主动校验‌：在关键决策前，让 Agent 用当前问题反向检验记忆中信息的时效性和适用性，降低因为过时记忆导致的行为漂移。

总结

让 Agent 拥有持续上下文与长期一致性，本质上是在‌记忆分层、状态持久化、上下文工程、状态合并控制、记忆生命周期管理‌这五个维度上搭建一套闭环的工程体系。它让 Agent 不再是一个每次对话都重新认识你的客服，而是一个能跨会话、跨任务、跨中断，始终保持对你的理解与任务执行连贯性的“持续工作者”。

ERC-8004：AI代理的链上信任层

ERC-8004的诞生，正是为了解决智能体网络中的信任赤字。它不是一个AI模型，而是专为AI代理设计的链上信任基础设施，本质上是AI代理的“去中心化工商局+大众点评+认证实验室”。

其技术核心由三大注册中心构成：

身份注册中心‌：这是基石。ERC-8004创新地使用ERC-721标准为每个代理分配唯一的AgentID。这意义深远——代理身份从标识符变成了可拥有、可交易的资本资产，这正是AgentFi的开端。

声誉注册中心‌：这是一个防垃圾邮件的反馈系统。服务提供方完成任务后，客户可以对交互进行反馈，这些评价数据会被记录在链上。如果行为数据能变成金融原语，那么声誉系统就不只是好评差评，而是Agent世界的信用评分——声誉分高的Agent可以获得更高的信用额度、更低的交易成本和优先的任务分配。

验证注册中心‌：当涉及高风险任务时，ERC-8004允许通过可插拔的信任模型进行密码学验证，包括零知识机器学习证明、TEE预言机或质押重执行等方式。用户可以为低风险任务选择声誉验证，为高风险任务要求密码学证明。

从生态价值看，ERC-8004的身份和声誉注册表只是基础数据层。真正的价值创造在于谁能在这个数据层之上建立Agent的信用评估和金融服务——Agent贷款、Agent保险、Agent信用额度，也就是整个金融服务栈。

# 2026-05-21
<!-- DAILY_CHECKIN_2026-05-21_START -->


















AI与Web3的结合是当前技术融合的热点，主要应用场景包括：

‌智能合约优化‌：AI可以对智能合约的代码进行自动审计，提前发现潜在的漏洞和风险，提升合约的安全性；同时，AI能根据网络运行数据，动态优化智能合约的执行逻辑，提升交易处理效率。

‌去中心化数据服务‌：AI算法可以在保障数据隐私的前提下，对链上数据进行分析挖掘，为用户提供个性化的数据服务；同时，AI参与去中心化存储网络的管理，智能调度存储资源，提升存储效率和数据安全性。

‌Web3应用智能化‌：在DeFi领域，AI可以分析市场行情、用户行为，为用户提供更精准的借贷、投资建议；在NFT领域，AI生成独特的数字艺术作品，同时利用AI技术实现NFT的真伪鉴定和价值评估。

‌DAO治理辅助‌：AI为DAO（去中心化自治组织）提供智能决策支持，通过分析社区成员的提案、投票数据，挖掘社区共识，辅助DAO进行更科学的治理决策；同时，AI可以自动执行DAO的日常事务，提升治理效率。
