# Week 1 综合进阶：设计一个受限 Web3 助手

> 提交人：2758star | 日期：2026-05-18

## 要解决什么问题

新手看区块链交易很懵。区块浏览器上一堆 hex 和地址，不知道这笔交易到底做了什么、转了什么东西、有没有风险。我的想法是：让 AI 读链上数据，翻译成人话，但必须把「链上能验证的事实」和「AI 自己猜的」分开——不能让 AI 胡说八道用户还以为是真的。

这个助手的**核心设计原则**就一条：只读不写。AI 可以读链上数据、解码事件、生成解释，但不能碰私钥、不能签名、不能发起交易。

## 输入和输出示例

### 输入

```bash
node index.js 0x5460f5cdff49a32806f301f6216c68f4ace75b57e2e1fc53b9b4320fcd0c7a50 --llm
```

一个 Sepolia 测试网的 tx hash。

### 输出（LLM 生成部分）

```
### 1. 这笔交易做了什么
发送方 0x30eB...9EB2 调用了合约 0xe205...272D 的 transfer 函数，
向 0x69eA...3B20 转入了 155 个代币。交易成功。

### 2. 涉及的资产和地址
| 角色 | 地址 | 资产变动 |
|------|------|----------|
| 发送方 | 0x30eB...9EB2 | 代币 -155 |
| 接收方 | 0x69eA...3B20 | 代币 +155 |
| 代币合约 | 0xe205...272D | 无变动 |

### 3. 事实 vs 推断
| 事实（链上可验证） | 推断（模型推测） |
|-------------------|-----------------|
| tx hash、区块高度、Gas 消耗都是链上数据 | 发送方可能是代币持有者 |
| Transfer 事件记录了从 A 到 B 转了 155 个代币 | 可能是支付、转账给朋友、或交易所充值 |
| Nonce 为 0，像是该地址第一笔交易 | 不能确定就是第一笔，可能之前有失败的 tx |

### 4. 不确定的部分
- 无法确定交易目的（为什么转这笔钱）
- 无法确定两个地址的归属（是谁的钱包）
- 代币元数据未读取成功时无法确定代币名称

### 5. 签署类似交易前应检查什么
1. 确认接收地址是否正确（一个字符错了就没了）
2. 确认代币合约地址不是假冒的
3. 确认转账金额（注意小数位数）
4. Gas 价格别设太低，不然一直卡着
5. 如果是 approve 交易要更小心——授权意味着对方可以动你的币
```

## 人工确认点

| 步骤 | 谁做 | 为什么 |
|------|------|--------|
| 读链上数据 | 工具自动 | 只读操作，无风险 |
| 解码事件和函数调用 | 工具自动 | 规则匹配，不需 AI |
| LLM 生成解释 | DeepSeek 自动 | AI 翻译链上数据 |
| **验证 LLM 输出是否准确** | **人工** | LLM 可能幻觉代币名称、猜测错误动机 |
| **判断地址归属** | **人工** | 链上数据不包含身份信息 |
| **确认代币信息** | **人工** | 去 Etherscan 验证合约是否 verified |
| **决定是否签署类似交易** | **人工** | 工具只解释历史，不预测未来 |

核心规则：AI 的输出永远是「参考」，不是「结论」。用户必须自己对照 Etherscan 验证。

## 风险与限制

1. **LLM 幻觉**：模型可能编造代币名称、猜测错误目的。缓解：所有输出标注来源（链上事实 vs 模型推断），链上事实必须在 Etherscan 可验证。

2. **事件解码不完整**：当前只解码了 ERC20 的 Transfer 和 Approval 事件，DeFi 协议的自定义事件（Swap、Stake、Borrow 等）无法识别。缓解：原始 calldata 和 topics 仍然输出，用户可以手动查询。

3. **代币元数据缺失**：部分合约不实现标准的 name/symbol/decimals 方法，或实现不标准。此时显示原始数值，提示用户去 Etherscan 查看。

4. **公共 RPC 限速**：用的免费 Sepolia RPC，请求多了可能被限。缓解：可替换为 Infura / Alchemy。

5. **不会提醒风险交易**：工具只解释交易内容，不会主动判断「这笔交易像骗局」。用户需要自己判断。

## 如何验证结果

1. 拿 tx hash 去 https://sepolia.etherscan.io 查，对比工具输出的 from/to/value/gas —— 这些必须完全一致
2. 看 Etherscan 的「Logs」标签页，和工具输出的「事件日志」对比
3. LLM 输出中「事实」栏的每一项，必须能在 Etherscan 上直接找到对应数据
4. 如果工具读了代币元数据（name/symbol/decimals），去 Etherscan 的合约页面验证

## 技术实现

- ethers.js v6 读 Sepolia 链上数据
- DeepSeek API 生成自然语言解释（可选，--llm 参数）
- 只做 eth_call / eth_getTransaction 等只读查询
- 代码在个人学习仓库 experiments/tx-interpreter/
